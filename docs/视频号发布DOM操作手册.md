# 微信视频号视频发布 — DOM 操作手册

## 一、页面架构（关键）

### 1.1 微前端架构

视频号助手页面（`channels.weixin.qq.com/platform`）采用腾讯「无界」(wujie) 微前端框架。页面分为两个层次：

- **外层文档（宿主）**：`https://channels.weixin.qq.com/platform/...`，包含左侧导航栏、页脚。这些元素在普通 DOM 中，可直接用 CSS 选择器访问。
- **子应用（微前端内容）**：运行在 wujie 创建的 **闭合 shadow DOM** 内的 **iframe** 中。子应用的 URL 路径前缀为 `https://channels.weixin.qq.com/micro/content/...`，包含右侧主内容区域（视频列表、发布表单等）。

### 1.2 对自动化的影响

由于 wujie 使用的是**闭合 shadow DOM**（`shadowRoot.mode = 'closed'`），普通的 `document.querySelector` 和 `element.shadowRoot` 无法穿透访问子应用内部的 DOM 元素。

**解决方案**：自动化工具需要通过 **frame（帧）** 机制访问子应用。在 Playwright 中：

1. 通过 `page.frames()` 获取页面上所有帧（包括 shadow DOM 内的 iframe）
2. 找到 URL 包含 `/micro/content/` 的帧
3. 在该帧上执行后续所有元素查找和操作

在 Selenium 中，可通过 `driver.switch_to.frame()` 切换到子应用帧。

### 1.3 如何识别当前所在的帧

- 外层文档的 URL：`https://channels.weixin.qq.com/platform/post/create`
- 子应用帧的 URL：`https://channels.weixin.qq.com/micro/content/post/create`
- 子应用的 API 请求路径前缀：`/micro/content/cgi-bin/mmfinderassistant-bin/`

当你在子应用帧中时，所有元素查找操作（CSS选择器、文本匹配等）都应在该帧上下文中执行。

## 二、登录流程

### 2.1 登录页面

- **URL**：`https://channels.weixin.qq.com/login.html`
- **判断是否已登录**：检查当前 URL 是否包含 `login`。如果包含，说明需要扫码。

### 2.2 扫码登录

登录页面显示一个微信二维码。定位方式：
- **文本匹配**：页面上有「微信扫码登录」或类似的提示文字
- **图片元素**：二维码图片，通常是一个 `<img>` 标签，src 为 base64 编码的二维码图片

### 2.3 登录态持久化

扫码成功后 URL 跳转为 `/platform` 或 `/platform/post/list`。保存浏览器的 Cookie / Storage State 可实现后续免登录。Cookie 有效期通常数天到数周。

## 四、视频发布表单页（核心）

### 4.1 页面信息

- **URL（外层）**：`https://channels.weixin.qq.com/platform/post/create`
- **URL（子应用帧内）**：`https://channels.weixin.qq.com/micro/content/post/create`
- **面包屑**：页面顶部显示 `视频管理 / 发表动态`

### 4.2 页面布局

页面分为左右两大区域：
- **左侧**：视频上传区域（约占 40% 宽度）
- **右侧**：信息填写表单（约占 60% 宽度）
- **底部**：操作按钮栏（三个按钮）

### 4.3 左侧：视频上传区域

| 元素 | 定位方式 | 说明 |
|------|---------|------|
| 上传区域（虚线框） | 文本匹配 `上传时长8小时内`，或区域内有 `+` 图标 | 点击触发文件选择对话框 |
| **隐藏的 file input** | `input[type="file"]` | **最可靠的上传方式**：直接设置此元素的 files，无需模拟点击 |
| 上传说明文字 | 包含文本 `上传时长8小时内，大小不超过20GB，建议分辨率720p及以上，码率10Mbps以内，格式为MP4/H.264格式` | 用于确认上传区域已加载 |
| 信息图标 | 上传说明文字旁的 `(i)` 图标 | 点击可能显示更多上传要求 |

**上传操作**：
- **优先方案**：定位 `input[type="file"]` 元素，直接设置文件（Playwright: `setInputFiles`，Selenium: `sendKeys`）
- **备选方案**：监听 `filechooser` 事件，然后点击上传区域触发文件选择对话框

**上传后的 DOM 变化**：
- 上传区域（虚线框 + `+` 图标）被替换为视频预览/进度信息
- 可能出现文字如「正在处理文件」「上传中」等进度提示
- 上传完成后，底部操作按钮从灰色禁用变为可点击状态

- 灰色/禁用状态：视频尚未上传完成，或必填项未填写

        ├ 监听文字「正在处理」「上传中」等出现后消失

        ├ 定位：placeholder 为「添加描述」的输入框
        └ 操作：点击 → 清空 → 输入描述文字

Step 9  （可选）在子应用帧中填写「短标题」
        ├ 定位：placeholder 包含「短标题」的 input
        └ 操作：点击 → 输入标题

Step 10 （可选）在子应用帧中设置「位置」
        ├ 定位：显示当前地名的下拉框（如文字「天津市」）

Step 11 （可选）在子应用帧中设置定时发表

        └ 操作：等待按钮变为橙色（可点击）→ 点击

Step 13 验证发布结果
        ├ URL 跳转回 /platform/post/list（视频列表页）→ 发布成功
        ├ 页面出现错误提示 → 发布失败

## 六、元素定位策略总结

### 6.1 最可靠的定位方式（按优先级排序）

1. **文本精确匹配**：对于有明确文字的按钮/标签，优先用文本匹配（如 `发表视频`、`保存草稿`）
2. **placeholder 属性**：对于输入框，用 placeholder 文本定位（如 `添加描述`、`短标题`）
3. **元素类型 + 属性**：`input[type="file"]` 用于文件上传
4. **标签文字 + 相邻元素**：用标签文字（如「视频描述」「短标题」）定位到 label，再找相邻的输入控件

### 6.2 不可靠的定位方式

1. **CSS class 名称**：页面使用动态生成的 hash 作为 class 名（如 `.abc123`），每次构建都可能变化，**绝对不要用 class 名定位**
2. **DOM 层级路径**：页面结构可能随版本更新变化，硬编码的 XPath 路径容易失效
3. **元素 ID**：大多数元素没有稳定的 ID 属性

### 6.3 需要特别注意的易混淆文字

| 容易搞混的 | 实际正确文字 | 出现位置 |
|-----------|-------------|---------|
| ~~发布视频~~ | **发表视频** | 视频列表页右上角的橙色按钮 |
| ~~发表动态~~ | **视频管理 / 发表动态** | 发布页面的面包屑标题（注意「发表动态」是面包屑的一部分，不是按钮名） |
| ~~发布~~ | **发表** | 发布表单底部的提交按钮 |
| ~~定时发布~~ | **定时** / **定时发表** | 「定时发表」是字段标签，「定时」是单选项文字 |

## 七、关键网络请求（供 API 级自动化参考）

以下是发布流程中浏览器发出的关键 API 请求，均在子应用帧中发出，路径前缀为 `/micro/content/cgi-bin/mmfinderassistant-bin/`：

| 阶段 | API 路径 | 方法 | 说明 |
|------|---------|------|------|
| 页面初始化 | `helper/helper_upload_params` | POST | 获取视频上传参数（分片大小、上传地址等） |
| 页面初始化 | `post/get_finder_post_comm_info` | POST | 获取发布通用信息 |
| 页面初始化 | `post/check_finder_comm_face` | POST | 人脸/身份验证检查 |
| 页面初始化 | `post/finder_author_get_post_verify_info` | POST | 发布验证信息 |
| 页面初始化 | `post/get-finder-post-trace-key` | POST | 获取追踪 key |
| 位置搜索 | `helper/helper_search_location` | POST | 搜索位置信息 |
| 合集列表 | `collection/get_collection_list` | POST | 获取可添加的合集列表 |
| 视频上传 | 由 `helper_upload_params` 返回的地址 | — | 实际视频文件上传（可能是分片上传） |

视频处理使用了 WebAssembly 编解码器：`rhino_video.wasm`（来自 `aladin.wxqcloud.qq.com`），用于浏览器端的视频预处理。

## 八、视频上传的技术要求

上传区域显示的说明文字完整内容：

> 上传时长8小时内，大小不超过20GB，建议分辨率720p及以上，码率10Mbps以内，格式为MP4/H.264格式

| 参数 | 要求 |
|------|------|
| 时长 | ≤ 8 小时 |
| 文件大小 | ≤ 20GB |
| 分辨率 | 建议 ≥ 720p |
| 码率 | ≤ 10Mbps |
| 格式 | MP4 / H.264 |
