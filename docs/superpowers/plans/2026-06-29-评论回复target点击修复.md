> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 修复上一个 PR 合并后仍然存在的两个症状 —— (1) `tryExpandMoreAndScroll` 写操作仍针对 tabs-content 容器，document 不动导致搜索卡在顶部反复 "回顶-小滚"； (2) 根评论找到后回复按钮在视口外时回退到 root 中心坐标，导致 `replyToComment` 整页搜 "回复" 按钮时点到错评论的输入框。

**Tech Stack:** TypeScript, patchright/playwright, CDP, Prisma, Jest.

## Task 1: 改 `tryExpandMoreAndScroll` 写操作直走页面滚动

**Files:**
- Modify: `apps/ts-api-gateway/src/crawlers/douyinCrawler.ts`（`tryExpandMoreAndScroll` 内 "滚动加载更多" 段，约 L5911-5938）
- Test: `apps/ts-api-gateway/src/crawlers/__tests__/scrollBounded.test.ts`（两个 `tryExpandMoreAndScroll` 的 it）

**根因：** 修复后 `tryExpandMoreAndScroll` **读** document（`cdpGetDocumentScrollState`）但**写** 仍走 `scrollCommentArea(state.clientHeight * 0.6)`，最终 `cdpSmartScroll` 用 tabs-content 选择器在 `findScrollContainer` 中命中（rect 50×100+ 满足）→ 调 `scrollInContainer` 滚 inner 容器，**document 不动**。结果：算法重复执行但页面无位移，搜索卡住，需多轮才能因 DOM 偶然变化找到目标，表现为"在顶部来回好几次"。

**Step 1: 改 `tryExpandMoreAndScroll` 的写操作**

将 `apps/ts-api-gateway/src/crawlers/douyinCrawler.ts:5935` 替换：

原代码（L5935-5936）：
```typescript
    await this.scrollCommentArea(page, sh.ch * 0.6);
    await HumanActions.wait(page, 1000, 1500);
    return true;
```

（注：当前文件里变量名是 `sh.ch * 0.6` 来自旧版本；新版本应是 `state.clientHeight * 0.6`。请按实际代码定位 — grep `state.clientHeight \* 0.6` 找到 "滚动加载更多" 段的最后两行。）

替换为：
```typescript
    // ★ 修复：直接调 cdpSmartScroll 走空 selectors 强制 scrollPage（页面滚动），
    // 避免 findScrollContainer 命中 tabs-content inner 容器导致 document 不动。
    await HumanActions.cdpSmartScroll(page, [], state.clientHeight * 0.6, 'down');
    await HumanActions.wait(page, 1000, 1500);
    return true;
```

`HumanActions` 已是 import（用于 `cdpGetDocumentScrollState`），无需新 import。

**Step 2: 改 `scrollBounded.test.ts` 的 `tryExpandMoreAndScroll` 测试**

文件中 `cdpSmartScrollMock` 与 `getScrollStateMock` 已在 mock 块顶部定义。两个 `tryExpandMoreAndScroll` 的 it（约 L88-105）当前断言 `crawler.scrollCommentArea` 被调用，需改为断言 `cdpSmartScrollMock` 被调用且 selectors 为空数组。

将两个 it 改写：

```typescript
describe('tryExpandMoreAndScroll reads document bottom', () => {
  it('returns false when document at bottom (not tabs-content)', async () => {
    getScrollStateMock.mockResolvedValue({ scrollY: 1747, clientHeight: 1305, scrollHeight: 3052 });
    const page = { evaluate: jest.fn().mockResolvedValue(null) } as any;
    const crawler = new DouyinCrawler('fp_test') as any;
    crawler.injectEsbuildPolyfill = jest.fn().mockResolvedValue(undefined);
    const result = await crawler.tryExpandMoreAndScroll(page, 1);
    expect(result).toBe(false); // 1747+1305=3052 >= 3052-10 → 到底
    expect(cdpSmartScrollMock).not.toHaveBeenCalled();  // 到底不滚
  });

  it('scrolls page (empty selectors) and returns true when not at bottom', async () => {
    getScrollStateMock.mockResolvedValue({ scrollY: 800, clientHeight: 1305, scrollHeight: 3052 });
    const page = { evaluate: jest.fn().mockResolvedValue(null) } as any;
    const crawler = new DouyinCrawler('fp_test') as any;
    crawler.injectEsbuildPolyfill = jest.fn().mockResolvedValue(undefined);
    const result = await crawler.tryExpandMoreAndScroll(page, 1);
    expect(result).toBe(true);
    // ★ 关键断言：走空 selectors 强制页面滚动（不走 inner 容器）
    expect(cdpSmartScrollMock).toHaveBeenCalledWith(
      page, [], expect.any(Number), 'down'
    );
  });
});
```

## Task 2: `findReplyBtnInContainer` 失败时 `scrollRootIntoView` 再重试，不回退到 root 中心

**Files:**
- Modify: `apps/ts-api-gateway/src/crawlers/douyinCrawler.ts`（`scrollExpandAndFindTarget` 内 level=1 分支，约 L5011-5020）
- 已有 `scrollRootIntoView(page, x, y)` 方法（L5349），无需新增

**根因：** `findReplyBtnInContainer` (L5299) 只返回视口内按钮（`r.top < vh`）。当根评论卡片很高、回复按钮在视口外时返回 null。回退路径返回 `rootMatch.x, rootMatch.y`（root 中心）→ `replyToComment` (L4248) 整页搜 "回复" 按钮挑**离 root 中心最近**的，最近的往往是顶部某条评论的回复按钮（target 按钮在视口外，top 评论按钮在视口内且离 root 中心距离相近甚至更近），导致误点 + 错输入框。

**Step 1: 改 `scrollExpandAndFindTarget` level=1 失败路径**

将 `apps/ts-api-gateway/src/crawlers/douyinCrawler.ts:5011-5020` 替换：

原代码：
```typescript
      // ── B. 如果目标就是根评论 → 直接返回该根评论的"回复"按钮坐标 ──
      if (target.level === 1) {
        const replyBtn = await this.findReplyBtnInContainer(page, rootMatch.containerSel);
        if (replyBtn) {
          logger.info({ elapsedMs: Date.now() - startT0 }, '[Reply::Find] Root reply btn located');
          return replyBtn;
        }
        logger.warn('[Reply::Find] Root found but reply btn missing');
        // 回退：返回根评论中心坐标
        return { x: rootMatch.x, y: rootMatch.y };
      }
```

替换为：
```typescript
      // ── B. 如果目标就是根评论 → 直接返回该根评论的"回复"按钮坐标 ──
      if (target.level === 1) {
        let replyBtn = await this.findReplyBtnInContainer(page, rootMatch.containerSel);
        if (!replyBtn) {
          // 按钮在视口外 → 滚 root 到视口中间再重试
          logger.warn('[Reply::Find] Root found but reply btn off-viewport, scrolling into view');
          await this.scrollRootIntoView(page, rootMatch.x, rootMatch.y);
          await HumanActions.wait(page, 300, 600);
          replyBtn = await this.findReplyBtnInContainer(page, rootMatch.containerSel);
        }
        if (replyBtn) {
          logger.info({ elapsedMs: Date.now() - startT0 }, '[Reply::Find] Root reply btn located');
          return replyBtn;
        }
        // 仍找不到：返回 null 让外层循环继续 tryExpandMoreAndScroll 加载更多，
        // 而不是回退到 root 中心导致 replyToComment 整页搜时点到错评论。
        logger.warn('[Reply::Find] Root found but reply btn still missing after scrollIntoView');
        return null;
      }
```
