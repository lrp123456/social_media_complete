# 登录链路与监控调度修复 — Design Spec

**Date:** 2026-06-29
**Status:** Draft
**Scope:** 修复登录卡片刷新失败、视频号二维码截图兜底、登录不复用、未登录调度四类问题。问题 1/3 优先，4 与 2 随后。

---

## 1. 背景

四个用户报告的 bug，根因均落在"登录流程 + 监控调度 + 浏览器窗口复用"链路：

### 1.1 问题 1：除视频号外卡片刷新返回"无法创建登录标签页"

`sm-ts-api` 日志证据（2026/6/29 13:46–13:54）：

```
13:46:18 [error] getBrowser: CDP 连接失败
13:48:54 发送消息: ❌ 无法创建登录标签页   (douyin F5刷新)
13:49:06 发送消息: ❌ 无法创建登录标签页   (kuaishou F5刷新)
13:49:19 发送消息: ❌ 无法创建登录标签页   (kuaishou 强制刷新)
13:54:51 发送消息: 🔄 已刷新 tencent 登录页  (视频号成功)
```

数据库证据（三账号共用同一窗口）：

| userId | platform | window_id | external_id | status |
|---|---|---|---|---|
| 6 | douyin | 4 | `68a259626bb2...` | login_required |
| 11 | kuaishou | 4 | `68a259626bb2...` | init |
| 13 | tencent | 4 | `68a259626bb2...` | login_required |

`wechatBotService.ts:787` 卡片"强制刷新/F5刷新" → `ensureLoginTab` → `getBrowser` → `connectOverCDP`。卡片操作时该窗口正被 kuaishou 监控任务占用（13:41–13:51 同窗口运行 `manual_..._11`），CDP 连接争抢失败 → `getBrowser` 返回 null → `ensureLoginTab` 返回 null → "无法创建登录标签页"。视频号成功仅因其操作时窗口空闲。

### 1.2 问题 2：视频号 QR 返回屏幕截图

视频号登录页实际 DOM：
```html
<div class="qrcode-wrap">
  <img class="qrcode" src="data:image/png;base64,...">
  <div class="mask"><div class="refresh-wrap">
    <div class="refresh-icon weui-icon-outlined-refresh"></div>
    <p class="refresh-tip">二维码已过期，点击刷新</p>
  </div></div>
</div>
```

`captureQR`（`loginTabRegistry.ts:197-229`）用 `qrSelectors=['.qrcode-img','img[src*="qr"]','canvas']`，三个全不匹配实际 `img.qrcode`（class 是 `qrcode` 非 `qrcode-img`；src 是 base64 data URI 不含 "qr"）→ 第 228 行 `falling back to full page screenshot` 返回整页截图。过期刷新选择器 `.qrcode-refresh` 同样不匹配视频号实际 `.refresh-wrap`。

### 1.3 问题 3：登录不复用原有网站

`ensureLoginTab`（`loginFlowHelpers.ts:96-101`）：`find` 找不到带 localStorage 标记的登录 tab 时，总是 `openLoginTab`（`ctx.newPage()` + `goto loginUrl`）新建标签页，**不复用 monitorService 已打开的平台 tab**。叠加问题 1 的 CDP 失败，连新建都做不到。

另有一独立 bug：`monitorService.sendLoginQR:451` 用 `String(user.windowId)`（数据库内键 BigInt）传给 `getBrowser`，而 `getBrowser` 期望 BitBrowser `externalId`。与 `wechatBotService` 用 `String(user.window.externalId)` 不一致 → 监控触发的登录走 fallback 截图。

### 1.4 问题 4：未登录仍按短间隔倒计时，probe 状态机未接入

`monitorService.ts:729-819` 存在 `triggerLoginProbe` 状态机：检测到未登录后按 30/60/120/240min 递退避探测、不发 QR、探测到 `logged_in` 自动恢复 `active`。但审查发现 **该状态机是死代码——从未被任何路径调用**：

- `monitorService` 内部 1045–1361 多处 `updateUserStatus(userId,'login_required')` + `sendLoginQR`，但**全程无 `setFlowState`/`triggerLoginProbe` 调用**
- 各 crawler（`kuaishouCrawler:278`、`douyinCrawler`、`tencentCrawler:203`）检测到未登录只调 `captureAndSendQR` 发 QR，不进状态机

结果：所有平台未登录后只发一次 QR，调度仍按 active/idle 模式（180–1200s）短间隔倒计时，无长间隔探测。日志中快手 11 `status=init` 且能检测到登录表单并发 QR，但状态未切 `login_required`，是未登录未被识别的典型表现。

## 2. 根因

| 问题 | 根因 |
|---|---|
| 1 | 三账号共用窗口，卡片登录与监控并发争抢 CDP；`getBrowser` 连接失败直接返回 null，无重试；`ensureLoginTab` 返回 null 上报"无法创建登录标签页" |
| 2 | 视频号 `qrSelectors` 与实际 `img.qrcode` 不匹配；过期刷新选择器与 `.refresh-wrap` 不匹配 |
| 3 | `ensureLoginTab` 总是 `openLoginTab` 新建 tab，不复用监控已开的平台 tab；`sendLoginQR:451` windowId 用数据库内键而非 externalId |
| 4 | `triggerLoginProbe` 状态机未接入任何检测路径；快手未登录检测后不发 `login_required` 状态切换 |

## 3. 设计决策

### 3.1 问题 1/3：复用监控 session/tab + getBrowser 加固

**决策方向**（用户已确认）：复用监控 session/tab。

#### 改动 A：`monitorService.sendLoginQR` windowId 修正

`monitorService.ts:417-451`：查 `platformAccount` 时 `select` 加 `window: { select: { externalId: true } }`，`String(user.windowId)` 改为 `String(user.window.externalId)`，与 `wechatBotService:774` 口径统一。

#### 改动 B：`ensureLoginTab` 复用优先

`loginFlowHelpers.ts:ensureLoginTab` 在 `find` 未命中后、`openLoginTab` 之前，新增"复用监控平台页"路径：

1. `getBrowser(windowId)` —— 复用 monitorService 已建立的 CDP session（`getBrowser:246-258` 已有按 `${windowId}_*` 遍历复用逻辑），不重新 `connectOverCDP`
2. 枚举 `browser.contexts()[0].pages()`，找 URL 含 `config.domain` 且**未带 `__login_tab_mark__` 标记**的平台页
3. 命中则 `page.goto(config.loginUrl)`（复用"原有网站"），写入 localStorage 标记，`register` 为登录 tab
4. 仅当无可用平台页时才 `openLoginTab` 新建

`getBrowser` 复用 session 消除并发 `connectOverCDP` 争抢。

#### 改动 C：`getBrowser` 连接失败重试

`browserManager.ts:259-274`：`connectOverCDP` 失败时，先调 BitBrowser `openWindow(windowId)`（`/browser/open`，已开则返回现有 WS）确保窗口运行，再重试一次 CDP 连接；仍失败才返回 null。当前直接返回 null 导致"无法创建登录标签页"。

### 3.2 问题 4：接入 probe 状态机 + 前端"恢复登录"按钮

#### 改动 D：新增统一未登录入口

在 `monitorService` 新增 `markLoginRequiredAndProbe(userId, platform, windowId, flowId)`：
1. `db.updateUserStatus(userId, 'login_required')`
2. `setFlowState(userId, flowId, { status:'login_required', cooldownLevel:0, cooldownUntil: Date.now()+30*60*1000, lastProbeAt:0 })`
3. 首次发 QR（调 `sendLoginQR` 或 `captureAndSendQR`）
4. `triggerLoginProbe(userId, platform, windowId, flowId)`

各 crawler / monitorService 检测到未登录处（`kuaishouCrawler:278`、`douyinCrawler`、`tencentCrawler:203`、`monitorService:1045-1361`）改为调用此入口，替代裸 `captureAndSendQR`。后续 30/60/120/240min 探测只 `checkLoginState`、不发 QR。

#### 改动 E：快手未登录状态切换

审查 `kuaishouCrawler` 登录检测分支（`[Login] Login required` 路径），确认检测到未登录后调用改动 D 的 `markLoginRequiredAndProbe`（含 `updateUserStatus('login_required')`），补齐当前缺失的状态切换。

#### 改动 F：admin-dashboard 平台卡片"恢复登录"按钮

- 在平台卡片组件（`apps/admin-dashboard/src/components/matrix/`）新增按钮，仅当 `status === 'login_required'` 时显示
- 后端新增 `POST /api/platform-accounts/:id/probe-login`，调用 `triggerLoginProbe` 的"立即探测、忽略 cooldown"变体
- 探测到 `logged_in` → 恢复监控；未登录 → 发 QR

### 3.3 问题 2：视频号 QR 捕获修正

#### 改动 G：视频号 qrSelectors

`selectors.json` 视频号 `loginFlows.creator.qrSelectors`：
```json
"qrSelectors": ["img.qrcode", ".qrcode-wrap img.qrcode", "canvas"]
```

#### 改动 H：视频号过期刷新选择器

视频号 flow 配置新增 `qrRefreshSelector: '.refresh-wrap'`；`captureQR` 过期检测（`loginTabRegistry.ts:184-195`）使用该配置选择器替代硬编码 `.qrcode-refresh`，命中过期 mask 先点刷新再截图。

## 5. 风险与回滚

- **问题 1/3**：`ensureLoginTab` 复用平台页需排除已带登录标记的 tab，避免误复用旧登录页。`getBrowser` openWindow 重试增加约 3–5s 延迟，可接受。回滚即还原 `ensureLoginTab` 直走 `openLoginTab`。
- **问题 4**：`triggerLoginProbe` 此前是死代码，接入后开始实际探测，需观察 Redis 状态键与 30min 递退避是否符合预期。回滚即各检测处还原为裸 `captureAndSendQR`。
- **问题 2**：纯选择器配置，风险低。回滚即还原 qrSelectors。
- 三组改动独立，可分别提交、分别验证。

## 6. 验收标准

1. 三账号共用窗口场景下，卡片"强制刷新/F5刷新"对 douyin/kuaishou/tencent 均不再返回"无法创建登录标签页"，能发送新 QR；监控任务运行期间卡片操作不因 CDP 争抢失败。
2. `ensureLoginTab` 优先复用监控已开的平台 tab，仅在无可用页时新建；`monitorService.sendLoginQR` 用 externalId 传 `getBrowser`。
3. 视频号 `captureQR` 返回 `img.qrcode` 裁剪截图，不再 fallback 全页；过期 mask 经 `.refresh-wrap` 点击刷新后重新截图。
4. 各平台检测到未登录后进入 `triggerLoginProbe` 状态机：首次发 QR，后续 30/60/120/240min 递退避探测不发 QR，探测到登录自动恢复 `active`。
5. 快手未登录被识别（status 切 `login_required`），不再以 `init` 短间隔倒计时。
6. admin-dashboard 平台卡片在 `login_required` 时显示"恢复登录"按钮，点击触发立即探测。
7. 收口规范：浏览器操作经 `HumanActions.*` / `LoginTabRegistry`，无新增裸 `page.evaluate`/`page.mouse`（QR 激活点击沿用 `captureQR` 现有 `page.mouse.click`，属既有模式）。
