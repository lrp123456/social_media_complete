# 评论回复 scrollCommentArea 抖动 + 按钮误点修复设计

**日期**: 2026-06-29
**状态**: 已批准
**分支**: fix/douyin-comment-reply-jitter-and-button (续)

## 背景

上一轮修复（3 commits: aa31dac, 4e627d8, 0135277）尝试解决两个 bug，但重新执行后：

- **Bug 1（滚动抖动）**：依旧未解决。日志显示 `scrollCommentArea('top')` `totalMs:9776`，仅 1 轮 `scrollY:0`，但 `cdpSmartScroll` 仍执行了完整 9.7 秒鼠标滚动动画。
- **Bug 2（点错评论）**：症状变形。目标评论 `lqq/很漂亮`，实际回复了 `lqq/22221`。`elementFromPoint` 回退命中了错误容器。

## Bug 1 根因

### 日志证据

```
round:1 direction:"top" scrollY:0 clientHeight:1305 scrollHeight:2041
msg:"[scrollCommentArea] reached top/bottom"
direction:"top" totalMs:9776
```

### 因果链

`scrollCommentArea('top')` 的 for 循环逻辑（douyinCrawler.ts:3583-3603）：

1. 进入 round 0 → 立即调用 `cdpSmartScroll(page, [], 3000, 'up')`
2. `cdpSmartScroll` 用空 selectors → `findScrollContainer` 返回 null → 走 `scrollPage`
3. `scrollPage` 执行鼠标移动 + CDP Input.dispatchMouseEvent（含 overshoot 抖动概率 0.15）
4. 多次 "Trying fallback CDP wheel scroll" → 总耗时 9.7 秒
5. 滚动完成后才读取 `cdpGetDocumentScrollState` → 发现 `scrollY:0` → "reached top" → break

**根因**：缺少前置边界检查。页面已在顶部时仍触发完整滚动动画，overshoot + 多次 fallback 造成 9.7 秒可见抖动。

## Bug 2 根因

### 日志证据

```
20.382s  [Reply::Find] Root comment located (scrollRound:2)
          → findRootCommentByUsernameContent 返回 rootMatch.y=1374
          → 但 viewportHeight=1305，y:1374 在视口下方

20.388s  [findReplyBtnInContainer] DIAG  containerFound:false
          → containerSel = 'div[class*="container-"]:nth-child(27)'

20.394s  [Reply::Find] Root found but reply btn off-viewport, scrolling into view
          → scrollRootIntoView(1385, 1374) → elementFromPoint(1385,1374) = null (视口外)
          → 回退到"最近容器"距离匹配 → 找到错误容器（"22221"）
          → 把"22221"滚到视口中央

20.868s  [findReplyBtnInContainer] DIAG  containerFound:false (第二次)
          → centerFallback = {x: vp.w/2, y: vp.h/2} ≈ (1385, 652)
          → elementFromPoint(1385, 652) → 命中"22221"容器
          → 找到"22221"的回复按钮 → 点击 → 回复了错误评论
```

### 因果链

**缺陷 1：`:nth-child(N)` 选择器设计性错误**

`findRootCommentByUsernameContent`（douyinCrawler.ts:5229/5316）拼接选择器：
```javascript
var containerSel = sels[si] + ':nth-child(' + (ci + 1) + ')';
```

- `containers = document.querySelectorAll('div[class*="container-"]')` 返回 DOM 树中所有匹配元素
- `ci` 是这个扁平列表中的索引（0, 1, ..., 26）
- `:nth-child(27)` 意为"父元素的第 27 个子元素"
- **`querySelectorAll` 的第 N 个匹配 ≠ 父元素的第 N 个子元素** — 两者是完全不同的概念
- 结果：`containerFound` 永远 false

**缺陷 2：`elementFromPoint` 在视口外返回 null**

`rootMatch.y = 1374 > viewportHeight = 1305` — 根评论中心在视口下方。

- `findReplyBtnInContainer` 的 `elementFromPoint(1385, 1374)` → 返回 null（坐标在视口外）
- `scrollRootIntoView` 的 `elementFromPoint(1385, 1374)` → 返回 null
- `scrollRootIntoView` 回退到"最近容器距离匹配" → 找到错误容器（"22221"）
- 错误容器被滚到视口中央
- 第二次 `findReplyBtnInContainer` 用视口中心 `elementFromPoint(1385, 652)` → 命中"22221"的回复按钮

**两个缺陷叠加**导致回复了错误的评论。

## 修复设计

### Bug 1 修复：`scrollCommentArea` 前置边界检查

**位置**：`douyinCrawler.ts` `scrollCommentArea` 方法，for 循环前（~line 3582）

**逻辑**：

在 for 循环之前，先调用 `cdpGetDocumentScrollState(page)` 读取当前滚动状态。如果已在边界：
- `direction === 'top'` 且 `scrollY ≤ 0` → 直接 `return true`，不进入循环
- `direction === 'bottom'` 且 `scrollY + clientHeight ≥ scrollHeight - 10` → 直接 `return true`

加日志 `[scrollCommentArea] already at top/bottom, skipping`。

**影响范围**：所有 5 个 `'top'/'bottom'` 调用点受益。`number` 入参路径不受影响（走独立的 `cdpSmartScroll` 分支）。

### Bug 2 修复：增强 `findRootCommentByUsernameContent`，同一次 evaluate 内返回回复按钮坐标

**方案**（用户已选方案 A）：在找到匹配容器的同一个 `page.evaluate` 内，直接查找该容器的回复按钮。消除 `:nth-child` 选择器和 `elementFromPoint` 回退。

#### 1. 增强 `findRootCommentByUsernameContent` 的返回值

现有返回值：
```typescript
{ x: number, y: number, containerSel: string, isExpanded: boolean, subReplyCountInPage: number }
```

增强后：
```typescript
{ x: number, y: number, containerSel: null, isExpanded: boolean, subReplyCountInPage: number,
  replyBtn: { x: number; y: number } | null }
```

#### 2. evaluate 内查找回复按钮的逻辑

在找到匹配容器后（username + text 匹配通过），在该容器的 DOM 子树内查找回复按钮：

```
1. 遍历 container.querySelectorAll('[class*="operations-"]')
2. 在每个 operations 区域内找 [class*="item-"]，text === '回复'
3. 检查 getBoundingClientRect()：width > 0 && height > 0 && top < vh
4. 如果找到 → replyBtn = { x: left + width/2, y: top + height/2 }
5. 如果没找到 → container.scrollIntoView({block:'center', behavior:'instant'}) → 重新查找
6. 如果重新查找后找到 → replyBtn = { x, y }
7. 如果仍未找到 → replyBtn = null
```

**关键点**：`scrollIntoView` 在 evaluate 内同步执行，`getBoundingClientRect()` 立即反映新位置。与现有 `scrollRootIntoView`（douyinCrawler.ts:5490）使用相同的 `scrollIntoView({block:'center', behavior:'instant'})` 模式，已验证可行。

#### 3. 改写 `scrollExpandAndFindTarget` level=1 路径

**现有流程**（douyinCrawler.ts:5022-5065）：
```
findRootComment → findReplyBtnInContainer(containerSel, fallback)
  → null → scrollRootIntoView(x, y) → findReplyBtnInContainer(containerSel, centerFallback)
```

**新流程**：
```
findRootComment (增强版，内含回复按钮查找)
  → replyBtn 有值 → 直接返回 { x: replyBtn.x, y: replyBtn.y }
  → replyBtn 为 null → scrollRootIntoView(x, y) + wait → 重新调用增强版 findRootComment
    → replyBtn 有值 → 返回
    → replyBtn 仍为 null → 返回 null（让外层循环继续 tryExpandMoreAndScroll）
```

#### 4. 消除的代码

| 代码 | 位置 | 操作 |
|------|------|------|
| `:nth-child(N)` 选择器拼接 | line 5229 (isAntiDetectionV2 分支), line 5316 (普通分支) | 删除，`containerSel` 设为 `null` |
| level=1 对 `findReplyBtnInContainer` 的调用 | line 5025, line 5034 | 删除，用 `rootMatch.replyBtn` 替代 |
| `elementFromPoint` 5x5 网格回退 | `findReplyBtnInContainer` 内 (line 5421-5428, 5450-5457) | level=1 不再触发；保留函数本体 |
| `centerFallback` 计算 | line 5032-5033 | 删除 |

#### 5. 保留的代码

| 代码 | 原因 |
|------|------|
| `findReplyBtnInContainer` 函数本体 | 保留，未来可能用于其他场景；level=1 不再调用 |
| `scrollRootIntoView` | 用于 `replyBtn=null` 的回退 + level=2 路径 |
| `expandRootRepliesIfNeeded` | 不变，仍用 `rootMatch.x, rootMatch.y` + `elementFromPoint`（level=2 风险较低，不在本次范围） |
| 诊断日志（DIAG） | 暂时保留便于观察 |
| `containerSel` 字段 | 设为 `null`，保留字段名以兼容 `expandRootRepliesIfNeeded` 的类型签名 |

#### 6. isAntiDetectionV2 双分支

`findRootCommentByUsernameContent` 有两个分支：
- `isAntiDetectionV2()` → `HumanActions.safeEvaluate` (line 5102-5246)
- 普通 → `page.evaluate` (line 5248-5338)

两个分支都需要增强：在返回匹配结果前，加入回复按钮查找逻辑。逻辑完全相同，只是 evaluate 的调用方式不同。

## 测试策略

更新 `scrollBounded.test.ts` 中 `scrollCommentArea` 的 2 个测试用例：

- **"top" 用例**：模拟 `cdpGetDocumentScrollState` 返回 `{ scrollY: 0, clientHeight: 1305, scrollHeight: 2041 }` → 断言 `cdpSmartScroll` **不被调用** → 断言返回 `true`
- **"bottom" 用例**：模拟 `cdpGetDocumentScrollState` 返回 `{ scrollY: 941, clientHeight: 1100, scrollHeight: 941+1100 }` → 断言 `cdpSmartScroll` **不被调用**

### Bug 2 测试

新增测试用例（在 `scrollBounded.test.ts` 或新文件）：

- **"replyBtn found in evaluate"**：模拟 `findRootCommentByUsernameContent` 返回 `replyBtn: {x: 100, y: 200}` → 断言 `findReplyBtnInContainer` **不被调用** → 断言 `scrollRootIntoView` **不被调用** → 断言返回 `{x: 100, y: 200}`

## 风险评估

| 风险 | 影响 | 缓解 |
|------|------|------|
| `scrollIntoView` 在 evaluate 内不生效 | 回复按钮仍找不到 | 回退到 `scrollRootIntoView` + 重试（已有逻辑） |
| 回复按钮需 hover 才显示 | evaluate 内找不到 | 当前日志证明按钮无需 hover 即可见；如未来变化，可加 hover 步骤 |
| level=2 路径仍有类似风险 | 子评论回复可能点错 | 不在本次范围，但增强后的 `findRootCommentByUsernameContent` 可未来扩展到 level=2 |
| `cdpGetDocumentScrollState` 返回 null | 前置检查失败 | 已有 null 检查，跳过前置守卫，进入正常循环 |
