# 登录注册表串号与监控并发修复 — Design Spec

**Date:** 2026-06-30
**Status:** Draft
**Scope:** 修复 6 个用户报告的回归 bug。核心根因是 `LoginTabRegistry` 注册表 key 缺 platform 维度导致三平台共用窗口时串号（覆盖 #1/#2/#5/#6），叠加同 (window,platform) 监控并发互斥缺口（#3）与超时过长（#4）。

---

## 1. 背景

上一份 spec（`2026-06-29-登录链路与监控调度修复-design.md`）修复了选择器与 `ensureLoginTab` 复用，但遗漏了一个系统性串号根因：三个平台的 `flowId` 都是 `"creator"`，三账号共用同一 `windowId`，而 `LoginTabRegistry` 的内存 key 是 `${windowId}:${flowId}` —— 三平台 key 完全相同，互相覆盖。`find` 内存命中只按 key 查、不校验 domain，于是 `captureQR` 拿到的是其他平台的 page。

### 1.1 问题 1/5/6：刷新返回错误截图（全屏或他平台）

`data/selectors.json` 确认：

| platform | flowId | domain | qrSelectors |
|---|---|---|---|
| douyin | `creator` | `creator.douyin.com` | `img[aria-label="二维码"]` 等 |
| kuaishou | `creator` | `cp.kuaishou.com` | `img[alt="qrcode"]` 等 |
| tencent | `creator` | `channels.weixin.qq.com` | `div.qrcode-wrap img` 等 |

注册表 key 全是 `${windowId}:creator`。`loginTabRegistry.ts:71` 内存命中后直接返回 record，**不校验 `record.domain`**。后果：

- 视频号 `ensureLoginTab` → `find('4','creator')` 命中最后注册的 record（可能是抖音 page）→ `captureQR(抖音page, 视频号config)` → 视频号 qrSelectors 在抖音页全不匹配 → `loginTabRegistry.ts:232` 全页兜底 → **发出抖音截图** = 问题 6
- 快手刷新命中错误 page → qrSelectors 不匹配 → 全页截图 = 问题 1；"初次加载正确"是因为注册表冷态走 `openLoginTab` 正确建页；"偶发可发送"是恰好命中快手自己的 record
- 抖音刷新偶尔命中错误 page = 问题 5

附证：localStorage 标记 `__login_tab_mark__`（`loginTabRegistry.ts:254`）只存 `flowId` 不存 `platform`，`find` 枚举兜底（`:92` `markData.flowId === flowId`）同样无法区分平台 —— 串号是系统性的。

### 1.2 问题 2：登录/刷新偶尔新建 tab 不复用

key 冲突下缓存态混乱：`find` 可能命中其他平台的 record（返回非 null），导致 reuse/openLoginTab 路径行为不确定；reuse 路径（`loginFlowHelpers.ts:114`）排除条件只看 `flowId` 不看 `platform`。

### 1.3 问题 3：视频号一直一阶段"等待中"，两个监控进程并发

用户确认实际表现：前端显示"阶段一 等待中"，且**同一视频号账号有两个监控进程同时在阶段一等待**，后端 page 此刻停在评论页而非作品管理页。

根因有二：
1. **并发互斥缺口**：`monitorService.ts:1840-1854` 调度去重依赖 per-window BullMQ 队列按 `userId` 去重，但存在入口绕过去重（probe-login 立即探测、前端"恢复登录"、手动触发），导致同 (window, tencent) 出现两个并发监控。
2. **Phase1 无 wrong-page 检测**：两个进程并发驱动同一 page，一个把页面导航到评论管理页，另一个在 Phase1 等作品管理页加载 → 永远等不到 → "等待中"挂住，不 fast-fail。

### 1.4 问题 4：无复用 tab 时空白页等很久 / 卡片刷新等很久

- `captureQR`（`loginTabRegistry.ts:204`）在错误 page 上对每个 qrSelector × 每个 frame 跑 `waitForSelector(timeout:8000)`，3–6 选择器 × 多 frame 累积 30–60s 才兜底 —— key 冲突的副作用放大。
- `openLoginTab`（`:252`）`goto(timeout:30000)` + `getBrowser` openWindow 重试 + 多处 `waitForTimeout(3000)`。

## 2. 根因

| 问题 | 根因 |
|---|---|
| 1/2/5/6 | `LoginTabRegistry` key `${windowId}:${flowId}` 缺 platform 维度，三平台 `flowId='creator'` + 共用 window → key 撞车 → `find` 返回错误平台 page → `captureQR` 选择器不匹配 → 全页兜底发错图；localStorage 标记同样缺 platform |
| 3 | 同 (window,platform) 监控并发互斥缺口（去重按 userId 而非 window+platform，部分入口绕过去重）+ Phase1 无 wrong-page fast-fail |
| 4 | `captureQR` 错误 page 上 8s×选择器×frame 累积数十秒（#1 副作用）+ `openLoginTab` goto 30s + 多处固定 3s 等待 |

## 3. 设计决策

### 3.1 改动 A：注册表 key 加 platform 维度（修 #1/#2/#5/#6）

**决策方向（用户已确认方案 A）**：根因最小根治。

1. `LoginTabRegistry` 所有方法签名加 `platform` 参数：`register/unregister/find/openLoginTab/closeLoginTab` 改为 `(windowId, platform, flowId, ...)`。key 改为 `${windowId}:${platform}:${flowId}`。
2. `openLoginTab`（`loginTabRegistry.ts:254`）localStorage 标记 JSON 增加 `platform` 字段。
3. `find` 内存命中后增加校验：`record.domain === config.domain`，不符则删缓存继续走枚举（防旧标记残留串号）。
4. `find` 枚举兜底（`:92`）从 `markData.flowId === flowId` 改为 `markData.platform === platform && markData.flowId === flowId`。
5. `ensureLoginTab` reuse 路径（`loginFlowHelpers.ts:114`）排除条件：带标记但 `markData.platform !== platform` 的 page 跳过。
6. 所有直接调用 `loginTabRegistry.*` 的位置需传 platform（oracle 核实后的真实清单）：

   | 文件 | 行号 | 方法 |
   |---|---|---|
   | `loginFlowHelpers.ts` | 99 | `find(windowId, flowId, browser, config.domain)` |
   | `loginFlowHelpers.ts` | 127 | `register(windowId, flowId, record)` |
   | `loginFlowHelpers.ts` | 139 | `openLoginTab(windowId, userId, flowId, browser, config)` |
   | `wechatBotService.ts` | 701 | `find(windowId, targetFlowId, browser, config.domain)` |
   | `wechatBotService.ts` | 744 | `closeLoginTab(windowId, targetFlowId)` |
   | `wechatBotService.ts` | 746 | `unregister(windowId, targetFlowId)` |
   | `monitorService.ts` | 781 | `find(windowId, fid, browser, config.domain)` |
   | `monitorService.ts` | 790 | `closeLoginTab(windowId, fid)` |
   | `monitorService.ts` | 792 | `unregister(windowId, fid)` |
   | **`xiaohongshuCrawler.ts`** | **1266** | **`register(xhsWindowId, 'mainsite', record)`** |

   `wechatBotService.ts:787/832` 实际调用的是 `ensureLoginTab`（间接），已由 `loginFlowHelpers.ts` 覆盖。

7. `LoginTabRecord` interface 加 `platform: Platform` 字段；`register` 写入；`find` 内存命中后用 `record.domain === config.domain` 校验（精确比较）—— **不再依赖 record 是否有 platform 字段**，但保留字段以便将来扩展。

`LoginFlowConfig` 已含 `domain`，`getLoginFlowConfig(platform, flowId)` 返回的 config 自带 platform 来源，无需改 `types.ts`。

**包加载机制澄清**（oracle 验证）：`packages/browser-core/package.json:6` 写 `"main": "./src/index.ts"`，运行时**直接加载 TS 源文件**（由 ts-node/tsx 编译），`dist/` 目录不存在。改动 A 改 `.ts` 后**立即生效，无需编译步骤**。

### 3.2 改动 B：监控 per-(window,platform) 互斥 + Phase1 fast-fail（修 #3）

1. **调度去重 key 升级**：`monitorService.ts:1846-1854` 去重从 `userId` 改为 `${windowId}:${platform}`，同窗口同平台只允许一个活跃监控任务。
2. **入口统一 + 内部去重**：所有监控入口（`wechatBotService.ts:729/754`、`matrix.ts:1227/1291` 的手动触发、前端"恢复登录"、probe-login）经 `enqueueMonitor` 调用。**`enqueueMonitor` 内部**实现 `${windowId}:${platform}` 去重（不仅依赖调度器层），防止"直接调 enqueueMonitor 绕过调度器层去重"。

3. **Phase1 wrong-page fast-fail**（oracle 关键反馈）：当前各平台 Phase1 入口的 URL 检测过于宽松，需补充**路径级规则**。仅靠域名（`creator.douyin.com` / `cp.kuaishou.com` / `channels.weixin.qq.com`）无法区分作品页与评论页——

   | 平台 | 正常 Phase1 URL 路径 | 被带偏 URL 路径 | 当前检测 |
   |---|---|---|---|
   | 视频号 | `/platform/post/list`（视频列表） | `/platform/comment/...`、`/platform/home` | `tencentCrawler.ts:974-978` 只检查 `/platform` 不含 `/login` —— 评论页仍含 `/platform`，**不会被拦截** |
   | 抖音 | `/creator-micro/home` 或 `/creator-micro/content/manage` | 评论管理路径 | `monitorService.ts:1006-1009` 只检查 `creator.douyin.com` 域名 |
   | 快手 | `/article/publish/video`（视频管理） | 评论管理路径 | `monitorService.ts:1180-1183` 只检查 `cp.kuaishou.com` 域名 |

   **实施要求**：
   - `runTencentCheck` Phase1 入口加：`currentUrl.includes('/post/list')` 或显式 `redirect to post list if not on it`
   - `runDouyinCheck` Phase1 入口加：`currentUrl` 路径在 `['/creator-micro/home', '/creator-micro/content/manage']` 中
   - `runKuaishouCheck` Phase1 入口加：`currentUrl` 路径含 `/article/publish/video`
   - 偏离时 `logger.warn` + 返回 `{phase:'Phase1', hasUpdate:false}` 并**释放 page 锁**

**probe-login 边界修正**（oracle 关键反馈）：原"排队等活跃监控完成"与 `force=true` 的"立即"语义矛盾，且逻辑上无意义（监控完成 = 已登录 = probe 无必要）。改为：
- `triggerLoginProbe` 入口检测：同 (window,platform) 有活跃监控任务 → **直接返回 `{probed:false, reason:'monitor_active'}`**，不排队
- `force=true` 不再特殊处理（"立即"语义被去重取代）
- 前端按钮"恢复登录"显示"`已有监控运行中`"toast

### 3.3 改动 C：时长收敛（修 #4）

1. `captureQR`（`loginTabRegistry.ts:204`）`waitForSelector` 8000ms→3000ms。
2. `captureQR` 循环前校验 `page.url()` 域名（**用 `new URL(page.url()).hostname` 与 `new URL(config.loginUrl).hostname` 做精确 hostname 比较**）—— 跨子域 OAuth 跳转（如 `open.weixin.qq.com` → `channels.weixin.qq.com`）期间 hostname 不匹配 → captureQR 快速返回 null（这是预期行为，OAuth 跳转页无 QR 码可截）。域名不符直接返回 null（不跑选择器、不兜底全页）—— 串号时快速失败而非发错图。
3. `openLoginTab`（`:252`）`goto` 30000ms→12000ms。
4. **`getBrowser` `connectOverCDP` timeout 独立评估**（oracle 关键反馈）：不盲目封顶 6s。当前 30s × 2 次 + 1s 间隔总时长约 61s；为时长收敛降到 ~2.5s 风险高（慢网络下 CDP 连接失败率上升）。建议改为：**保持 `connectOverCDP` timeout 30s 现状，但加重试 2 次（共 3 次尝试）+ 间隔 500ms 渐进延长**，总时长仍可能达 90s+。或**灰度观察**：先在测试环境跑 1 周 P95 失败率，再决定是否收紧。改动 C-4 暂不实现，仅记录为后续优化项。

改动 A 修好后，错误 page 路径整体消失，#4 长等待自然大幅减少；改动 C-1/C-2/C-3 是把残留固定等待压低。C-4 暂不实施。

## 5. 风险与回滚

- **改动 A**：`LoginTabRegistry` 是 `@social-media/browser-core` 包导出，改签名是 breaking change，10 个直接调用点需同步（已 oracle 核实清单完整）。`LoginTabRecord` type 加 `platform` 字段需同步更新 record 创建点。旧 localStorage 标记无 `platform` 字段，`find` 枚举兜底对 `markData.platform === platform` 判定为 `undefined !== platform` 自然被忽略，安全。回滚即还原 key 格式与方法签名。
- **改动 B**：去重从 userId 改为 window+platform，可能影响"同窗口不同平台本应并行"的场景 —— 现有 per-window BullMQ concurrency=1 本就串行，window+platform 去重是更严格的子集，不会比现状更松。`probe-login` 行为变化（不再排队，直接返回 `monitor_active`）是语义改变：前端"恢复登录"按钮在监控活跃时显示"已有监控运行中"toast，需前端配合。回滚即还原 userId 去重 + probe-login 排队逻辑。
- **改动 C**：`captureQR` URL 域名校验用 `new URL().hostname` 精确比较；跨子域 OAuth 跳转（`open.weixin.qq.com` → `channels.weixin.qq.com`）期间 hostname 不匹配 → captureQR 快速返回 null（预期行为，OAuth 跳转页无 QR 码可截）。`waitForSelector` 8s→3s 在慢网络下可能不够，灰度观察。`connectOverCDP` timeout 暂不动（独立评估中），回滚仅需还原 timeout 与移除 hostname 校验。
- 三组改动独立，可分别提交、分别验证。

## 6. 验收标准

1. **单元测试级别**（不可 E2E，三平台共用窗口在 CI 不可行）：三平台注册同旧 key 模拟串号 → 旧行为下 `find` 返回错误平台 record；新 key + domain 校验下 `find` 返回正确平台 record。视频号发 `img.qrcode` 裁剪截图（沿用上一批 7f4641a 选择器）。
2. **单元测试级别**：`find` 在 key 冲突场景下按 `record.domain === config.domain` 校验拒绝串号。
3. **单元测试级别**：mock `enqueueMonitor` 同 (window, tencent) 二次入队 → 断言去重拒绝。mock Phase1 在评论页 → 断言 fast-fail 返回。**集成测试**：上一批 PR（4e82a81/7d051e5/38f8ab8）测试套件重跑通过。
4. **定量目标**（oracle 反馈补充）：卡片刷新 P95 耗时 ≤15s（当前最坏路径 30-60s）；单次 `captureQR` 在错误 page 上 ≤3s 返回（域名校验前置）；`openLoginTab` 失败时 ≤12s 退出（goto 12s timeout）。
5. **单元测试级别**：`captureQR` 在 page URL 域名不符 `config.domain` 时返回 null，不跑选择器、不兜底全页；同域回调（`open.weixin.qq.com` → `channels.weixin.qq.com`）放行。
6. **回归测试**：上一批 4 个 PR 的测试套件（`monitorService.test.ts:37-65`、`loginFlowHelpers.test.ts`、`captureQRSelectors.test.ts`、`matrix.ts probe-login 路由`）全部通过。
7. 收口规范：浏览器操作经 `HumanActions.*` / `LoginTabRegistry`，无新增裸 `page.evaluate`/`page.mouse`（QR 激活点击沿用 `captureQR` 既有 `page.mouse.click`，属既有模式）。
