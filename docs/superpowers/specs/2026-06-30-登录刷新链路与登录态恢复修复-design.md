# 登录刷新链路与登录态恢复修复 — 设计

> 日期：2026-06-30
> 前置：`2026-06-30-登录注册表串号与监控并发修复` 已落地（提交 462dd76→e2318d1）。本 spec 处理该修复后残留/新暴露的三个独立 bug 集群。

## 1. 问题陈述

修复注册表 platform 维度后，日志（容器 `sm-ts-api`，2026-06-30 04:10–04:16）与用户复现暴露三个独立 bug：

### Bug #1a — 登录流程全链路都会新建标签页，无法复用
- 现象：用户报告"登录流程中包括前端在未登录状态点击更新、点击校验登录状态、在卡片点击刷新等等这些操作时，很多情况下都会新建一个标签页"。快手尤其严重（"新建了很多标签页，为什么还是无法复用"）。
- 涉及入口（均最终走 `ensureLoginTab`→`find`→`openLoginTab`）：
  1. 前端未登录点"更新" → `/monitor/accounts/:userId/trigger` → `enqueueMonitor` → 监控任务登录失败 → `sendLoginQR`（`monitorService.ts:1080/1249/1397/1621`）→ `ensureLoginTab`
  2. 前端点"恢复登录" → `/monitor/accounts/:userId/probe-login` → `triggerLoginProbe` → `find`（`monitorService.ts:801`）
  3. 企微告警点"强制刷新"/"F5刷新" → `wechatBotService.ts:787/832` → `ensureLoginTab`
- 日志实证：多次 `强制刷新 11 kuaishou creator` / `F5刷新 11 kuaishou creator`，每次都走 `openLoginTab` 新建，`find` 从未命中。

### Bug #1b — 快手刷新后没点开 QR 码 → 找不到 QR → 全屏兜底 / 纯文字告警 / F5 超时
- 现象：① 快手刷新后发全屏截图（非 QR）；② 偶发纯文字告警"🔐 快手 需要登录"无图片；③ `F5刷新失败: page.reload: Timeout 30000ms exceeded`。
- **核心（用户明确纠正）**：根因是刷新后**没有点击打开 QR 码、找到 QR 码再获取发送**，而不是全屏兜底本身有问题——**全页兜底保留，是合理的最终保险**。
- 日志实证：
  ```
  [ensureLoginTab] clicked QR switch "div.platform-switch" for kuaishou   ← 激活了 QR
  captureQR: img[alt="qrcode"] not found in any frame                      ← 但 QR 又没了
  captureQR: img[src*="data:image/"] not found in any frame
  captureQR: canvas not found in any frame
  captureQR: falling back to full page screenshot                          ← 兜底（保留）
  captureQR: full page fallback failed: page.screenshot: Timeout 30000ms exceeded.
  ```
  `page.reload: Timeout 30000ms exceeded. Call log: waiting for navigation until "domcontentloaded"`

### Bug #2 — 视频号"恢复登录"无效 + "更新"成功后仍显示需重新登录
- 现象：① 前端点"恢复登录"（probe-login）视频号仍无法恢复；② 点"更新"数据更新成功，但卡片仍显示"需重新登录"。

## 2. 根因分析

### Bug #1a 根因：复用/查找的 URL 筛选用 `domain`（监控域），快手登录域与监控域不同；内存丢失后无法恢复

`data/selectors.json` 三平台配置：

| 平台 | `domain`（监控域） | `loginUrl` hostname（登录域） | 同域？ |
|---|---|---|---|
| tencent | `channels.weixin.qq.com` | `channels.weixin.qq.com` | ✓ |
| douyin | `creator.douyin.com` | `creator.douyin.com` | ✓ |
| **kuaishou** | **`cp.kuaishou.com`** | **`passport.kuaishou.com`** | **✗** |

复用与查找逻辑全部用 `domain` 字段去筛页面 URL：

- `apps/ts-api-gateway/src/services/loginFlowHelpers.ts:108`（`ensureLoginTab` 复用分支）：`if (!url.includes(config.domain)) continue;`
- `packages/browser-core/src/loginTabRegistry.ts:91`（`find` 枚举兜底）：`if (!url.includes(domain)) continue;`

`find` 的复用分两层：
1. **内存命中**（`loginTabRegistry.ts:71`）：按 `${windowId}:${platform}:${flowId}` key 查 + `:76` 的 `cached.domain === domain` 校验。快手 record 存的 `domain=cp.kuaishou.com`（`openLoginTab:278` 存 `config.domain`），与传入 `domain` 一致 → **内存命中可复用**。
2. **枚举兜底**（`:88-112`）：内存 miss 时扫描所有页面 localStorage 标记恢复。但 `:91` 用 `url.includes(domain)` 筛 → 快手登录页 URL `passport.kuaishou.com` 不含 `cp.kuaishou.com` → **永远跳过** → 枚举永远返回 null。

后果：**只要内存注册表丢失（服务重启、tab 被手动关闭、跨进程、内存命中后被 `:80` 清理），快手已开的登录 tab 就再也找不到** → `ensureLoginTab:139` 走 `openLoginTab` 新建。这解释了"很多情况下都会新建"。视频号/抖音同域，枚举兜底能命中，故复用正常。

三入口（trigger→sendLoginQR、probe、强刷/F5）都经 `find`，故全链路都受影响。

### Bug #1b 根因：二次导航丢失 QR 激活 + 重新激活用非 CDP 点击不可靠

`wechatBotService.ts` 两条刷新路径在 `ensureLoginTab`（已点击 `div.platform-switch` 激活 QR，日志 `[ensureLoginTab] clicked QR switch` 印证）之后又执行二次导航，把刚激活的 QR 重置回密码登录页：

- `:789` 强制刷新：`await record.page.goto(config.loginUrl, { waitUntil: 'domcontentloaded', timeout: 30000 });`
- `:835` F5刷新：`await record.page.reload({ waitUntil: 'domcontentloaded', timeout: 30000 });`

随后 `captureQR`（`loginTabRegistry.ts:161-196`）尝试重新激活 QR，但激活用的是 `page.mouse.click`（非 anti-detection v2 的 CDP 点击），在 v2 模式下不可靠 → QR 没被点开 → 三个 `qrSelectors` 全 miss（日志印证）→ 走全页兜底（`:249-255`，**保留**）。

快手登录页 `domcontentloaded` 不稳定 + 全页截图对重定向中的页面超时 → 全页兜底 `page.screenshot` 抛 `Timeout 30000ms` → `captureQR` 返回 null → `sendLoginAlert(undefined)` → 纯文字告警。F5 `reload` 本身也 30s 超时。

而初次加载走 `sendLoginQR`→`ensureLoginTab`→`captureQR`，**不二次导航**，激活的 QR 保留 → 正常（日志 `04:00:53` 与 `04:10:52` 两次 `captured via "img[alt=\"qrcode\"]" (180x180)` 成功）。

**核心修复方向**：刷新后必须重新点击 `div.platform-switch` 打开 QR 码、确认 QR 可见后再截图；不要二次导航把激活冲掉；激活统一用可靠方式。

### Bug #2 根因：监控成功不重置 active + probe 找不到 tab 时不检测

`monitorService.ts` 四平台 `run*Check` 在 `handleLogin` 返回 true（登录有效）后继续抓取，但**从不把 `status` 重置为 `active`**：
- `runDouyinCheck:1017`、`runKuaishouCheck:1198/1215`、`runXiaohongshuCheck:1360`、`runTencentCheck:1582/1589`

账号一旦因风控/过期被置 `login_required`，即使后续监控登录成功并抓到评论，`status` 仍停留在 `login_required` → 卡片持续显示"需重新登录"（现象②）。

`triggerLoginProbe`（`:767-844`）在 `find` 返回 null 时（`:801-805`）直接 `delFlowState` 返回，不打开登录页也不检测 → 无残留登录 tab 时"恢复登录"对视频号无效（现象①）。

## 3. 设计方案（方案 A：根因修复 + 抽取共享激活逻辑）

### A0 — 复用/查找的 URL 筛选改用 `loginUrl` 的 hostname（修 #1a 全链路新建标签页）

将 `domain` 字段语义在代码注释中澄清为"监控域"，复用与查找的 URL 筛选改用 `new URL(config.loginUrl).hostname`，使登录域与监控域不同的平台（快手）在内存丢失后也能通过枚举兜底命中已开的登录 tab。

**改动点：**
- `loginFlowHelpers.ts:108`（`ensureLoginTab` 复用分支）：`if (!url.includes(config.domain)) continue;` → 改为基于 `loginUrl` hostname 的精确比较（hostname 相等或为 `.{hostname}` 子域）。
- `loginTabRegistry.ts:91`（`find` 枚举兜底）：同样改用 `loginUrl` hostname。`find` 签名第 5 参 `domain` 改名为 `loginHost`，调用方传入 `getLoginHost(config)`。
- `loginFlowHelpers.ts:99`（`ensureLoginTab` 内 `find` 调用）、`wechatBotService.ts:701`、`monitorService.ts:801`：传 `getLoginHost(config)`。

**辅助函数**（避免在多处重复 hostname 解析）：
```ts
// loginFlowHelpers.ts 导出
export function getLoginHost(config: LoginFlowConfig): string {
  try { return new URL(config.loginUrl).hostname; } catch { return config.domain; }
}
export function isOnLoginDomain(url: string, loginHost: string): boolean {
  try {
    const h = new URL(url).hostname;
    return h === loginHost || h.endsWith('.' + loginHost);
  } catch { return false; }
}
```

> **内存命中校验（`:71-81`）——关键修正**：当前 `:76` 是 `cached.domain === domain`。`cached.domain` 存的是 `config.domain`（监控域，`openLoginTab:278` 与复用分支 `:124` 均存 `config.domain`）。若简单地把传入参数从 `domain` 改为 `loginHost`，快手会因 `cached.domain`(cp.kuaishou.com) `!==` `loginHost`(passport.kuaishou.com) 而**误清内存命中**（`:80`），反而破坏复用。故内存命中校验**不读 `cached.domain` 字段**，改为校验 page 当前 hostname 在登录域：
> ```ts
> // find 内存命中（替换 :76）
> if (!cached.page.isClosed()) {
>   try {
>     if (isOnLoginDomain(cached.page.url(), loginHost)) return cached;
>   } catch { /* page 引用过期 */ }
> }
> this.tabs.delete(key);
> ```
> `LoginTabRecord.domain` 字段语义保持为监控域（`config.domain`）不变，监控侧既有使用不受影响。内存命中与枚举统一用 page 实际 hostname 判断，避免 `record.domain` 语义混淆。

### A1 — 抽取共享 `activatePlatformQR`，刷新后重新点开 QR 码（修 #1b 核心）

**核心（用户纠正）**：刷新后必须重新点击 `div.platform-switch` 打开 QR 码、确认 QR 可见后再截图。将散落在 `ensureLoginTab`（`loginFlowHelpers.ts:144-193` 平台 QR 切换 + 小红书预激活）与 `captureQR`（`loginTabRegistry.ts:161-196` `qrActivationSelector` 激活）的激活逻辑抽成一个共享函数 `activatePlatformQR(page, platform, config)`，统一使用 `HumanActions.cdpClick`（anti-detection v2）/ `page.mouse.click` 兜底，并**等待 QR 选择器可见**才返回。

```ts
// loginFlowHelpers.ts 导出
export async function activatePlatformQR(page: any, platform: string, config: LoginFlowConfig): Promise<boolean> {
  // 1. 平台 QR 切换（快手 div.platform-switch 等）
  const switchSelector = PLATFORM_QR_SWITCH[platform];
  if (switchSelector) {
    try {
      if (isAntiDetectionV2()) {
        if (await HumanActions.cdpIsElementVisible(page, switchSelector)) {
          await HumanActions.cdpClick(page, switchSelector);
          await new Promise(r => setTimeout(r, 2000));
        }
      } else {
        const el = await page.$(switchSelector);
        if (el) { await el.click(); await new Promise(r => setTimeout(r, 2000)); }
      }
    } catch { /* switch may not be needed */ }
  }
  // 2. qrActivationSelector 通用激活（原 captureQR:161-196 逻辑）
  if (config.qrActivationSelector) { /* ...原逻辑，统一 CDP... */ }
  // 3. 小红书预激活（原 ensureLoginTab:164-193 逻辑）
  if (platform === 'xiaohongshu') { /* ...原逻辑... */ }
  // 4. 等待任一 qrSelectors 可见
  for (const sel of (config.qrSelectors || [])) {
    try { await page.waitForSelector(sel, { timeout: 3000, state: 'visible' }); return true; } catch { continue; }
  }
  return false;
}
```

- `ensureLoginTab` 导航完成后调用 `activatePlatformQR`（替换原 `:144-193` 内联逻辑）。
- **强刷/F5 二次导航后也调用 `activatePlatformQR`**（关键：替代"依赖 captureQR 重新激活"）。这是修 #1b 的核心——二次导航冲掉了 `ensureLoginTab` 的激活，必须在导航后重新点开 QR。
- **`sendLoginQR:439-449` 短路分支**（page 已在登录域，直接 `captureQR` 不经 `ensureLoginTab`）：`captureQR` 调用前也必须先 `activatePlatformQR(page, platform, config)`，否则该路径 QR 未点开会截不到。这是 A1 的第三个调用点，原 spec 风险栏"sendLoginQR 走 ensureLoginTab 已激活，无需改"只覆盖 `:453` 路径，漏了此短路分支。
- `captureQR` 不再重复激活（移除 `:161-196` 的 `qrActivationSelector` 分支），只负责截图。

### A2 — 强刷/F5 导航 timeout 收敛 + 容错（修 #1b 超时）

- `wechatBotService.ts:789` `goto` / `:835` `reload`：`timeout: 30000` → `12000`。
- 用 try/catch 包裹导航，导航超时不中止整条刷新（继续走 `activatePlatformQR` + `captureQR`）。

### A3 — （删除，全页兜底保留）

按用户明确纠正，全页兜底（`loginTabRegistry.ts:249-255`）是合理的最终保险，**保留不动**。`captureQR` 选择器全 miss 时仍走全页兜底。纯文字告警问题由 A1（重新激活让 QR 出现，截到真 QR）+ A2（导航不再 30s 超时）解决，而非删除兜底。

### B1 — 监控成功后重置 `active` + 清 flowState（修 #2 现象②）

> **设计修正（grep 核实后）**：`runDouyinCheck`/`runXiaohongshuCheck` 无显式 `handleLogin` 调用（登录检测藏在 crawler 内部与风控分支），无法在"handleLogin 成功后"统一加。改为在 dispatch 入口（`monitorService.ts:976-987`）统一包装：`run*Check` 返回 `riskDetected === false` 即说明未触发风控/登录失败 → 登录有效 → 重置 active。`riskDetected === true` 时 `run*Check` 内部已 `updateUserStatus('login_required'/'risk_control')`，不重置。

抽 helper `markLoginRecovered(userId, platform)`：
```ts
// monitorService.ts
async function markLoginRecovered(userId: number, platform: string): Promise<void> {
  const { prisma } = await import('../lib/prisma');
  await prisma.platformAccount.update({
    where: { id: userId },
    data: { status: 'active', cooldownUntil: BigInt(0) },
  });
  // 清该平台所有 flowState
  const { getFlowIdsForPlatform } = await import('./loginFlowHelpers');
  for (const fid of getFlowIdsForPlatform(platform)) {
    await delFlowState(userId, fid);
  }
}
```
（`delFlowState` 已在 `:756` 导出，可见。）

dispatch 入口 `:976-987` 改为：
```ts
switch (task.platform) {
  case 'douyin': {
    const r = await runDouyinCheck(page, task, onProgress);
    if (!r.riskDetected) await markLoginRecovered(task.userId, 'douyin');
    return r;
  }
  case 'kuaishou': {
    const r = await runKuaishouCheck(page, task, onProgress);
    if (!r.riskDetected) await markLoginRecovered(task.userId, 'kuaishou');
    return r;
  }
  case 'xiaohongshu': {
    const r = await runXiaohongshuCheck(page, task, onProgress);
    if (!r.riskDetected) await markLoginRecovered(task.userId, 'xiaohongshu');
    return r;
  }
  case 'tencent': {
    const r = await runTencentCheck(page, task, onProgress);
    if (!r.riskDetected) await markLoginRecovered(task.userId, 'tencent');
    return r;
  }
  default:
    throw new Error(`不支持的监控平台: ${task.platform}`);
}
```

> 边角：wrong-page fast-fail 返回 `riskDetected=false`（如 `:1037`/`:1602`），此场景登录态未知。但 fast-fail 仅并发导航冲突时触发（罕见）且任务会重试，误重置 active 的代价 < 持续显示"需重新登录"的代价，可接受。`runTencentCheck` 第二个 handleLogin（`:2248`，重连路径）同在 `runTencentCheck` 调用链内，由 dispatch 统一覆盖。

### B2 — `triggerLoginProbe` 在 `find` null 时回退检测（修 #2 现象①）

`monitorService.ts:801-805` 当前：
```ts
const record = await loginTabRegistry.find(windowId, platform, fid, browser, config.domain);
if (!record) {
  await delFlowState(userId, fid);
  return;
}
```
改为 `find` null 时回退 `ensureLoginTab` 打开/复用登录页，再 `checkLoginState`：
```ts
let record = await loginTabRegistry.find(windowId, platform, fid, browser, getLoginHost(config));
if (!record) {
  const { ensureLoginTab } = await import('./loginFlowHelpers');
  record = await ensureLoginTab(windowId, userId, platform, fid);
  if (!record) { await delFlowState(userId, fid); return; }
}
const result = await loginTabRegistry.checkLoginState(record.page, config);
if (result === 'logged_in') {
  await markLoginRecovered(userId, platform);   // 复用 B1 helper
  // closeOnLoginSuccess 则关闭 tab，否则 unregister（同原 :809-813 逻辑）
  ...
} else {
  // 未登录 → 发 QR + 设 flowState 冷却（同原 :824-838 逻辑）
  // sendLoginQR 是本文件 :415 顶层导出函数，直接调用，无需 import
  await sendLoginQR(record.page, userId, platform, fid);
  ...setFlowState...
}
```

## 5. 风险与回滚

- **A0 改 `find` 签名**：`find` 第 5 参 `domain`→`loginHost`，3 处调用点（`loginFlowHelpers:99`、`wechatBotService:701`、`monitorService:801`）+ 2 处测试同步改。`packages/browser-core` 直接以 TS 源码被引用（`"main": "./src/index.ts"`），无需编译 dist。
- **A0 内存校验改用 page hostname**：`cached.page.url()` 在 page 已导航离开登录域时会让内存命中失效，回退枚举——语义正确（离开登录域的 tab 不应复用为登录 tab）。
- **A1 抽函数**：`captureQR` 移除激活分支后，所有调用方必须保证调用前已激活；三处补齐：`ensureLoginTab`（`:144-193` 替换）、强刷/F5（`:789/835` 导航后）、`sendLoginQR` 短路分支（`:441` 前）。`sendLoginQR:453` 走 `ensureLoginTab` 已激活，无需改。`captureQR` 的 QR 过期检测/刷新逻辑（`:201-217`）保留。
- **A1 强刷/F5 二次导航后重新激活**：`goto`/`reload` 会重置页面状态，`activatePlatformQR` 必须在导航**之后**调用；若导航超时（A2 容错），页面可能仍是旧页，`activatePlatformQR` 在旧页激活也可能成功（QR 还在）。
- **全页兜底保留**：A3 已删除，兜底不动。若 A1 激活仍偶发失败，兜底发全屏图——可接受（最终保险），且日志记录哪个 selector miss 便于排查。
- **B1 立即重置 active**：风控分支会覆盖，无副作用。
- **B2 probe 回退 `ensureLoginTab`**：可能打开新 tab；但配合 A0 修复后快手也能复用，新 tab 概率降低。`force=true` 路径（probe-login 路由）语义不变。

回滚：每个改动组（A0/A1/A2/B1/B2）独立提交，可单独 revert。

## 6. 验收标准

1. **#1a 全链路复用**：
   - 快手：连续两次 `强制刷新 11 kuaishou creator`，日志第二次出现 `[ensureLoginTab] reused existing platform tab`，不再出现 `openLoginTab: navigating`。浏览器标签页数量不增加。
   - 前端未登录点"更新"触发监控登录失败发 QR 时，第二次不再新建登录 tab（复用已有）。
   - 服务重启后（内存注册表清空），快手已开的登录 tab 仍能被 `find` 枚举命中（A0 修复）。
2. **#1b 快手刷新发 QR**：`强制刷新`/`F5刷新` 快手后，企微收到 QR 图片（非全屏）；日志出现 `activatePlatformQR` 激活 + `captureQR: captured via "img[alt=\"qrcode\"]"`。F5 不再 30s 超时（≤12s 容错）。偶发激活失败时走全页兜底（保留），不再纯文字告警（因 A2 导航不再超时、A1 重新激活让 QR 大概率出现）。
3. **#2 现象②**：视频号 `login_required` 状态下点"更新"，监控登录成功并抓到评论后，卡片 `status` 变 `active`，"需重新登录"消失。
4. **#2 现象①**：视频号无残留登录 tab 时点"恢复登录"，probe 回退打开登录页检测；若已登录则 `active`，未登录则发 QR。
5. 回归：抖音/视频号/小红书原有刷新与监控链路不受影响（同域平台 A0 行为不变）。
