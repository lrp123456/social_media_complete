# 评论回复 scrollCommentArea 抖动 + 按钮误点修复设计

> 日期: 2026-06-29
> 分支: fix/douyin-comment-reply-jitter-and-button
> 前置 commit: aa31dac, 4e627d8, 0135277（已合入分支但未解决根本问题）

## 背景与问题

### Bug 1: scrollCommentArea 抖动

`scrollCommentArea('top')` 在页面已处于顶部时仍执行 `cdpSmartScroll([], 3000, 'up')`，触发约 9.7 秒的鼠标滚动动画（含多次 CDP wheel scroll fallback + overshoot），造成可见的上下来回抖动。

**日志证据**：
```
round:1 direction:"top" scrollY:0 clientHeight:1305 scrollHeight:2041
msg:"[scrollCommentArea] reached top/bottom"
direction:"top" totalMs:9776
```

仅 1 轮，scrollY 已为 0，但 totalMs 仍达 9776ms — 因为 `cdpSmartScroll` 在检查 scrollY 之前就执行了完整滚动动画。

### Bug 2: 回复按钮点击错误评论

目标评论 `lqq/"很漂亮"`（level=1），实际回复了 `lqq/"22221"`（同用户名不同内容）。

**日志证据**：
```
20.382s  [Reply::Find] Root comment located scrollRound:2
20.388s  [findReplyBtnInContainer] DIAG  containerFound:false  ← :nth-child(27) 失败
20.394s  [Reply::Find] Root found but reply btn off-viewport, scrolling into view
20.401s  [Reply::Find] Root scrolled into view  x:1385 y:1374  ← rootMatch 原始坐标
20.868s  [findReplyBtnInContainer] DIAG  containerFound:false  ← 仍然失败
20.872s  [Reply::Find] Root reply btn located  elapsedMs:5192  ← 找到了错误评论的按钮
21.034s  [Reply] Target located, clicking reply  x:406 y:743  ← 点击了"22221"的回复按钮
```

## 根因分析

### Bug 1 根因

`scrollCommentArea` 第 3583-3603 行的循环结构：

```
for (round = 0; round < MAX_ROUNDS; round++) {
    cdpSmartScroll([], 3000, dir)     // ← 先滚动
    wait(200-500ms)
    state = cdpGetDocumentScrollState()
    if (scrollY <= 0) break           // ← 后检查
}
```

即使 scrollY 已为 0，第一轮仍执行 `cdpSmartScroll`，触发 9.7 秒滚动动画。缺少前置边界检查。

### Bug 2 根因（4 步连锁失败）

**缺陷 1: `:nth-child(N)` 选择器设计性错误**

`findRootCommentByUsernameContent` 第 5229 行：
```javascript
var containerSel = sels[si] + ':nth-child(' + (ci + 1) + ')';
```

- `ci` 是 `document.querySelectorAll('div[class*="container-"]')` 返回的扁平索引
- `:nth-child(N)` 的语义是「父元素的第 N 个子元素」
- 两者不等价：第 27 个 `querySelectorAll` 匹配 ≠ 任何父元素的第 27 个子元素
- 结果：`containerSel` 永远无法匹配 → `containerFound: false`

**缺陷 2: `elementFromPoint` 在视口外返回 null**

`findRootCommentByUsernameContent` 返回 `rootMatch.y = 1374`，但 `viewportHeight = 1305`。
- 根评论中心在视口下方（部分可见但中心超出）
- `elementFromPoint(1385, 1374)` 返回 null（坐标在视口外）

**缺陷 3: `scrollRootIntoView` 距离回退找到错误容器**

`elementFromPoint` 返回 null 后，`scrollRootIntoView` 回退到「最近容器距离匹配」：
```javascript
var containers = document.querySelectorAll('div[class*="container-"]');
// 找距离 (1385, 1374) 最近的容器
```
最近的容器不是"很漂亮"而是"22221" → 把"22221"滚动到视口中央。

**缺陷 4: 第二次 `findReplyBtnInContainer` 用视口中心命中错误容器**

```javascript
const centerFallback = { x: Math.round(vp.w / 2), y: Math.round(vp.h / 2) };
replyBtn = await this.findReplyBtnInContainer(page, rootMatch.containerSel, centerFallback);
```
视口中心现在是被错误滚来的"22221" → `elementFromPoint(centerX, centerY)` 命中"22221"容器 → 找到"22221"的回复按钮 → 点击错误评论。

## 修复设计

### Fix 1: scrollCommentArea 前置边界检查

**文件**: `apps/ts-api-gateway/src/crawlers/douyinCrawler.ts`
**位置**: `scrollCommentArea` 方法，第 3582 行 for 循环之前

在 `'top'/'bottom'` 分支的 for 循环前，先读取 `cdpGetDocumentScrollState`。如果已在边界，直接 `return true`，不触发任何滚动动画。

```typescript
// 前置边界检查：已在边界时跳过滚动，避免不必要的动画抖动
const initialState = await HumanActions.cdpGetDocumentScrollState(page);
if (initialState) {
  if (direction === 'top' && initialState.scrollY <= 0) {
    logger.info({ direction, scrollY: 0 }, '[scrollCommentArea] already at boundary, skipping');
    return true;
  }
  if (direction === 'bottom' && initialState.scrollY + initialState.clientHeight >= initialState.scrollHeight - 10) {
    logger.info({ direction, scrollY: initialState.scrollY }, '[scrollCommentArea] already at boundary, skipping');
    return true;
  }
}
```

**影响范围**: 所有 5 个 `'top'/'bottom'` 调用点（lines 2795, 3631, 4112, 4988, 5108）。不影响 `number` 入参路径。

### Fix 2: 增强 findRootCommentByUsernameContent，同一次 evaluate 内返回回复按钮坐标

**文件**: `apps/ts-api-gateway/src/crawlers/douyinCrawler.ts`
**位置**: `findRootCommentByUsernameContent` 方法（两个分支：isAntiDetectionV2 + 普通）

#### 返回值变更

现有：
```typescript
{ x: number; y: number; containerSel: string; isExpanded: boolean; subReplyCountInPage: number }
```

增强后：
```typescript
{
  x: number; y: number;
  containerSel: string;      // 保留但 level=1 路径不再使用
  isExpanded: boolean;
  subReplyCountInPage: number;
  replyBtn: { x: number; y: number } | null;  // 新增
}
```

#### evaluate 内逻辑变更

在找到匹配容器后（username + text 匹配成功且 `rect.top < vh && rect.bottom > 0`），增加回复按钮查找：

1. 遍历该容器的 `[class*="operations-"]` → `[class*="item-"]`，找 `textContent.trim() === '回复'` 且 `width > 0 && height > 0 && top < vh` 的按钮
2. 如果找到 → 设置 `replyBtn = { x: r.left + r.width/2, y: r.top + r.height/2 }`
3. 如果未找到（按钮可能在视口外）→ `container.scrollIntoView({ block: 'center', behavior: 'instant' })` → 重新查找 → 设置 `replyBtn` 或保持 `null`
4. 返回包含 `replyBtn` 的完整对象

**关键点**：`scrollIntoView` 在 evaluate 内使用容器 DOM 引用（非坐标），保证滚动的是正确容器。`getBoundingClientRect()` 在 `scrollIntoView` 后同步返回更新后的位置。

#### 调用方变更（scrollExpandAndFindTarget level=1 路径）

**位置**: 第 5022-5065 行

现有流程：
```
findRootComment → findReplyBtnInContainer(containerSel, rootXY)
  → null → scrollRootIntoView(rootXY) → findReplyBtnInContainer(containerSel, centerXY)
```

新流程：
```
findRootComment (增强版，内含回复按钮查找 + scrollIntoView)
  → replyBtn 有值 → 直接返回 { x: replyBtn.x, y: replyBtn.y }
  → replyBtn 为 null → scrollRootIntoView(x, y) → wait → 重新调用增强版 findRootComment → 返回 replyBtn
```

具体改写：
```typescript
if (target.level === 1) {
  const rootMatch = await this.findRootCommentByUsernameContent(page, target, rootContainerSels);
  // ... null check ...
  
  if (rootMatch.replyBtn) {
    logger.info({ elapsedMs: Date.now() - startT0 }, '[Reply::Find] Root reply btn located (inline)');
    return rootMatch.replyBtn;
  }
  
  // replyBtn 为 null：scrollIntoView 在 evaluate 内已尝试但失败，走外层重试
  logger.warn('[Reply::Find] Root found but reply btn null after inline scrollIntoView, retrying');
  await this.scrollRootIntoView(page, rootMatch.x, rootMatch.y);
  await HumanActions.wait(page, 300, 600);
  
  const retryMatch = await this.findRootCommentByUsernameContent(page, target, rootContainerSels);
  if (retryMatch?.replyBtn) {
    logger.info({ elapsedMs: Date.now() - startT0 }, '[Reply::Find] Root reply btn located (retry)');
    return retryMatch.replyBtn;
  }
  
  // 仍失败：返回 null
  logger.error({ containerSel: rootMatch.containerSel }, '[Reply::Find] Root reply btn still missing after retry');
  return null;
}
```

#### 消除的代码

| 代码 | 位置 | 原因 |
|------|------|------|
| `:nth-child(N)` 选择器拼接 | 2 处 evaluate 内 | 设计性错误，永远无法匹配 |
| level=1 路径对 `findReplyBtnInContainer` 的调用 | lines 5025, 5034 | 被增强版 evaluate 替代 |
| `elementFromPoint` 5x5 网格回退 | findReplyBtnInContainer 内 | level=1 不再调用此函数 |
| `centerFallback` 视口中心回退 | line 5033 | 不再需要 |

#### 保留的代码

| 代码 | 原因 |
|------|------|
| `findReplyBtnInContainer` 函数本体 | 保留，未来可能用于其他场景 |
| `scrollRootIntoView` | 用于 replyBtn=null 的外层重试 + level=2 路径 |
| `expandRootRepliesIfNeeded` | level=2 路径不变 |
| 诊断日志（DIAG） | 暂时保留便于观察 |
| `containerSel` 字段 | 保留为 null，兼容 level=2 的 `expandRootRepliesIfNeeded` |

### 不在本次修复范围

- `expandRootRepliesIfNeeded` 中的 `elementFromPoint` 模式（level=2 路径）— 风险较低，level=2 场景未触发
- `cdpSmartScroll` 性能优化（9.7s 单次调用）— 前置检查已避免不必要的调用
- `scrollRootIntoView` 的距离回退逻辑 — 增强版 evaluate 内的 `scrollIntoView` 已减少对外层 `scrollRootIntoView` 的依赖
