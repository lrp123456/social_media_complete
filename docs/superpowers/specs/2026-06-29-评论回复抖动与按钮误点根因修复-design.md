# 评论回复 scrollCommentArea 抖动 + 按钮误点修复设计

日期: 2026-06-29
分支: fix/douyin-comment-reply-jitter-and-button (已有 3 commit, 本次在其上继续)

## 背景

前 3 次 fix commit (`aa31dac` / `4e627d8` / `0135277`) 未能解决两个 bug:

- **Bug 1**: `scrollCommentArea('top')` 进入评论详情页时上下来回抖动
- **Bug 2**: 回复按钮点错评论 — 应回复 `lqq` 的"很漂亮"，实际回复了 `lqq` 的"22221"

## 根因分析

### Bug 1: scrollCommentArea 无前置边界检查

`scrollCommentArea` 的 `'top'/'bottom'` 分支在 for 循环内**先滚动再检查**:

```typescript
for (let round = 0; round < SCROLL_MAX_ROUNDS; round++) {
    await HumanActions.cdpSmartScroll(page, [], SCROLL_BOUNDED_PX, dir);  // ← 先滚动
    await new Promise(r => setTimeout(r, 200 + Math.random() * 300));
    const state = await HumanActions.cdpGetDocumentScrollState(page);
    if (direction === 'top' && state.scrollY <= 0) { break; }  // ← 后检查
}
```

即使 `scrollY` 已经是 0 (已在顶部), `cdpSmartScroll([], 3000, 'up')` 仍执行完整的鼠标滚动动画:
- `scrollPage` 移动鼠标到视口中心
- 多次 CDP `Input.dispatchMouseEvent` (wheel scroll)
- overshoot 行为 (15% 概率反向滚动)
- 内部 fallback 重试

日志证据: `totalMs:9776`, 仅 1 轮, `scrollY:0` — 9.7 秒滚动动画完全多余。

### Bug 2: :nth-child(N) 选择器设计性错误 + elementFromPoint 坐标超出视口

**缺陷 1: `:nth-child(N)` 选择器永远不匹配**

`findRootCommentByUsernameContent` (line 5229/5316):
```typescript
var containerSel = sels[si] + ':nth-child(' + (ci + 1) + ')';
```

`ci` 是 `document.querySelectorAll('div[class*="container-"]')` 返回的扁平数组索引。
但 `:nth-child(N)` 的语义是"父元素的第 N 个子元素"。
这两个概念完全不同 — 第 27 个 `querySelectorAll` 匹配 ≠ 任何父元素的第 27 个子元素。

日志证据: `containerSel: "div[class*=\"container-\"]:nth-child(27)"`, `containerFound: false` (每次都 false)。

**缺陷 2: elementFromPoint 坐标超出视口 → 距离回退找到错误容器**

`findRootCommentByUsernameContent` 返回 `rootMatch.y = 1374`, 但 `viewportHeight = 1305`。
根评论中心在视口下方 (部分可见: `rect.top < 1305 && rect.bottom > 0` 通过了可见性检查, 但中心 y=1374 在视口外)。

连锁失败链:
1. 第一次 `findReplyBtnInContainer`: `:nth-child(27)` 失败 → `elementFromPoint(1385, 1374)` 返回 null (y 在视口外) → 返回 null
2. `scrollRootIntoView(1385, 1374)`: `elementFromPoint(1385, 1374)` 返回 null → 距离回退找到最近的错误容器 ("22221") → `scrollIntoView({block:'center'})` 把"22221"滚到视口中央
3. 第二次 `findReplyBtnInContainer`: `centerFallback = {x: vp.w/2, y: vp.h/2}` ≈ (1385, 652) → `elementFromPoint(1385, 652)` 命中被错误滚到中心的"22221"容器 → 找到"22221"的回复按钮
4. 点击 (406, 743) → 回复了"22221"而非"很漂亮"

## 修复设计

### Fix 1: scrollCommentArea 前置边界检查

**文件**: `apps/ts-api-gateway/src/crawlers/douyinCrawler.ts`
**位置**: `scrollCommentArea` 方法, `'top'/'bottom'` 分支, for 循环前 (line ~3582)

在 for 循环前加前置检查:
```typescript
const initialState = await HumanActions.cdpGetDocumentScrollState(page);
if (initialState) {
    if (direction === 'top' && initialState.scrollY <= 0) {
        logger.info({ direction, scrollY: initialState.scrollY },
            '[scrollCommentArea] already at boundary, skipping scroll');
        return true;
    }
    if (direction === 'bottom'
        && initialState.scrollY + initialState.clientHeight >= initialState.scrollHeight - 10) {
        logger.info({ direction, scrollY: initialState.scrollY },
            '[scrollCommentArea] already at boundary, skipping scroll');
        return true;
    }
}
```

**影响范围**: 所有 5 个 `'top'/'bottom'` 调用点 (lines 2795, 3631, 4112, 4988, 5108)。不影响 `number` 入参路径。

### Fix 2: 增强 findRootCommentByUsernameContent, 同一次 evaluate 内返回回复按钮坐标

**文件**: `apps/ts-api-gateway/src/crawlers/douyinCrawler.ts`

#### 2a. 增强返回值类型

现有:
```typescript
{ x: number; y: number; containerSel: string; isExpanded: boolean; subReplyCountInPage: number }
```

增强后:
```typescript
{
    x: number; y: number;
    containerSel: string;  // 保留但不再用于 level=1 查找, 仅供 level=2 的 expandRootRepliesIfNeeded
    isExpanded: boolean;
    subReplyCountInPage: number;
    replyBtn: { x: number; y: number } | null;  // ★ 新增: 同一次 evaluate 内找到的回复按钮坐标
}
```

#### 2b. evaluate 内逻辑增强

在 `findRootCommentByUsernameContent` 的 evaluate 函数体中, 找到匹配容器后 (现有 username + text 匹配逻辑不变), 增加:

```typescript
// 在匹配容器内查找回复按钮
function findReplyBtnInEl(container: Element): { x: number; y: number } | null {
    var vh = window.innerHeight;
    var opsAreas = container.querySelectorAll('[class*="operations-"]');
    for (var oi = 0; oi < opsAreas.length; oi++) {
        var items = opsAreas[oi].querySelectorAll('[class*="item-"]');
        for (var ri = 0; ri < items.length; ri++) {
            if ((items[ri].textContent || '').trim() === '回复') {
                var r = items[ri].getBoundingClientRect();
                if (r.width > 0 && r.height > 0 && r.top < vh) {
                    return { x: Math.round(r.left + r.width / 2), y: Math.round(r.top + r.height / 2) };
                }
            }
        }
    }
    return null;
}

// 先直接查找
var replyBtn = findReplyBtnInEl(container);

// 如果没找到 (可能在视口外), scrollIntoView 后重试
if (!replyBtn) {
    (container as HTMLElement).scrollIntoView({ block: 'center', behavior: 'instant' });
    replyBtn = findReplyBtnInEl(container);
}
```

返回值增加 `replyBtn` 字段。`containerSel` 仍拼接 `:nth-child(N)` (保留供 level=2 使用, 但 level=1 不再依赖它)。

**关键**: 两个 evaluate 分支 (`isAntiDetectionV2()` + 普通分支) 都需增强。

#### 2c. 改写 scrollExpandAndFindTarget level=1 路径

**位置**: lines 5022-5065

现有流程:
```
findRootComment → findReplyBtnInContainer(containerSel, fallback)
  → null → scrollRootIntoView → findReplyBtnInContainer(containerSel, centerFallback)
```

新流程:
```typescript
if (target.level === 1) {
    const rootMatch = await this.findRootCommentByUsernameContent(page, target, rootContainerSels);
    // ... rootMatch null 处理不变 ...

    // ★ 如果增强版 evaluate 已找到回复按钮, 直接返回
    if (rootMatch.replyBtn) {
        logger.info({ elapsedMs: Date.now() - startT0 }, '[Reply::Find] Root reply btn located (in-evaluate)');
        return rootMatch.replyBtn;
    }

    // ★ replyBtn 为 null: scrollRootIntoView + 重新调用增强版 evaluate
    logger.warn('[Reply::Find] Reply btn not found in-evaluate, scrolling into view and retrying');
    await this.scrollRootIntoView(page, rootMatch.x, rootMatch.y);
    await HumanActions.wait(page, 300, 600);

    const retryMatch = await this.findRootCommentByUsernameContent(page, target, rootContainerSels);
    if (retryMatch?.replyBtn) {
        logger.info({ elapsedMs: Date.now() - startT0 }, '[Reply::Find] Root reply btn located (after retry)');
        return retryMatch.replyBtn;
    }

    // 仍找不到: 诊断日志 + return null (现有逻辑)
    logger.error({ containerSel: rootMatch.containerSel }, '[Reply::Find] Root found but reply btn still missing');
    return null;
}
```

**消除的调用**:
- `findReplyBtnInContainer` 的 2 处调用 (lines 5025, 5034)
- `elementFromPoint` 5x5 网格回退 (findReplyBtnInContainer 内, level=1 不再触发)

**保留的代码**:
- `findReplyBtnInContainer` 函数本体 — 保留, 不删除 (避免影响可能的未来调用)
- `scrollRootIntoView` — 保留, 用于 `replyBtn=null` 回退和 level=2 路径
- `expandRootRepliesIfNeeded` — 不变 (level=2 路径, 不在本次范围)
- 诊断日志 (DIAG) — 暂时保留

## 不在本次范围

- `expandRootRepliesIfNeeded` (level=2) 仍使用 `elementFromPoint(rootMatch.x, rootMatch.y)` — 同样的坐标超出视口风险存在, 但用户当前 bug 仅涉及 level=1

- `scrollRootIntoView` 的距离回退逻辑 — 保留现有实现, 因增强版 evaluate 内的 `scrollIntoView` 大幅降低了对它的依赖
