# 登录态检测与恢复按钮修复 Design

> 日期：2026-06-30
> 范围：快手登录态误判、抖音监控 0s 假成功、恢复登录按钮语义、快手/视频号滥发登录卡片
> 关联前序计划：`docs/superpowers/plans/2026-06-30-登录刷新链路与登录态恢复修复.md`（A0/A1/A2/B1/B2，已覆盖登录 tab 复用与 QR 激活）

## 背景与根因

### Bug 1 — 快手扫码登录后仍判未登录
`triggerLoginProbe` 与 `wechatBotService` "继续监控"指令在 `passport.kuaishou.com` 登录页上检测登录态。快手是唯一登录域（passport）≠ 监控域（cp）的平台：
- `LoginTabRegistry.checkLoginState` 用 `loggedOutIndicators`（含 `[class*="login"]`），登录页永远命中 → 永远 `logged_out`，于是反复探测/刷新登录页。
- "继续监控"指令 `wechatBotService.ts:711` 把 page `goto(config.loginUrl)`（passport 登录页）再判断 → 仍在登录页 → 误判未登录。

用户确认：登录成功后进入 `https://cp.kuaishou.com/profile`，右上角用户状态栏 `.user-info-dpd`（`//*[@id="app"]/div[1]/header/section[2]/div[4]/span/div[2]`）存在即已登录；登录失效时该 URL 会回退到登录页。且 `.user-info-dpd` 在任意 cp 页面都存在，不必每次都导航到 profile。

### Bug 2 — 抖音监控 0s 假成功
日志确认每轮 ~40ms 返回 success。`runDouyinCheck:1048-1059` 入口判断 `currentUrl.includes('creator.douyin.com')` 为 true 就跳过 `navigateToCreatorHome`，但 page 停在上一轮退出策略残留子页（`/creator-micro/data/following/follower`、`/content/original_protection`）→ `dyOk`（含 `/creator-micro/home` 或 `/creator-micro/content/manage`）不命中 → fast-fail 直接 return，被 `handleJob` 当 success 记账。

### Bug 3 — 恢复登录按钮不生效
前端"恢复登录"按钮（`page.tsx:1704-1720`）→ `POST /probe-login` → `triggerLoginProbe(force=true)` 异步探测，成功才 `markLoginRecovered`；且 `markLoginRecovered`（`monitorService.ts:764`）只做 DB 重置 + 清 flowState，**不调** `scheduleNext`，恢复后无倒计时。视频号同源问题导致"恢复不了"。

用户要求：按钮改为**直接置已登录 + 启动倒计时**，不检测、不发卡片。

### Bug 4 — 快手滥发登录卡片
`runKuaishouCheck:1236` 每轮 Phase 0 调 `ks.handleLogin`，后者在判 false 时 `page.goto(passport 登录页)` + `botManager.sendLoginAlert()` 发 QR 卡片 + **阻塞 5 分钟等扫码** + QR 过期重发 3 次。`checkLoginStatus` 误判 false（URL 含 `/passport` 直接 return false，或 sidebar 选择器漏命中）会触发每轮自动调度都发卡片 + 卡住 5 分钟。

用户规则：只有①监控阶段发现登录态失效 ②用户点"刷新" 才发卡片。自动轮询倒计时触发的监控发现失效 → 发一张卡片是**正确的**；不该有的是 handleLogin 的主动跳登录页 + 阻塞循环 + QR 过期重发。

## 设计

### 第 1 节：快手登录检测 + 发卡片策略（Bug 1 + Bug 4）

**A. 检测函数 `detectKuaishouLogin(page)`**（替换 `checkLoginStatus` 的误判逻辑）
- 不导航，直接看当前页是否存在右上角用户信息模块 `.user-info-dpd`。
- 命中且 `isVisible` → 已登录；否则 → 未登录。
- 修掉"URL 含 `/passport` 直接判 false"和"sidebar 选择器漏命中"两个误判源。

**B. `runKuaishouCheck` Phase 0 改造**（移除 `handleLogin` 阻塞）
- `detectKuaishouLogin(page)` 已登录 → 继续 Phase1。
- 未登录 → `db.updateUserStatus(login_required)` + `sendLoginQR(page, userId, 'kuaishou')`（发**一张**企微卡片）+ `return`。**不 goto 登录页、不阻塞 5 分钟、不 QR 过期重发**。
- 发卡片后该账号被 `getAllActiveUsers` 排除出调度，不再每轮重复发。

**C. `ks.handleLogin` 拆分**
- 保留 `captureAndSendQR` + QR 截图逻辑供"强制刷新/F5刷新"路径复用。
- `handleLogin` 不再被监控轮询调用（改为只检测不阻塞的薄封装，或标记 deprecated）。
- "继续监控"指令 `wechatBotService.ts:711` 的 `goto(config.loginUrl)` 改为 `goto('https://cp.kuaishou.com/profile')` + 看 `.user-info-dpd`：命中 → `markLoginRecovered` + `resetSchedulerTimer`；未命中 → 仍走 15min 重试。

**D. 视频号同步**
- `tencentCrawler.handleLogin`（`:149`）同样移除阻塞等待循环，改为检测失效→发卡片→return。
- `runTencentCheck:1610` / `:2269` 调用点同步：失效只发一张卡片 + 标 login_required + return，不阻塞。

**发卡片白名单最终对齐**：
- ✅ 监控阶段（自动轮询/手动更新）发现失效 → 发一张
- ✅ 用户点"强制刷新"/"F5刷新" → 发
- ❌ 不再因 handleLogin 阻塞循环反复发、不再 QR 过期自动重发 3 次

### 第 2 节：抖音 fast-fail 0s（Bug 2）

`runDouyinCheck:1048-1059` 入口改为"是否在工作页"作为导航判据：
```
const dyUrl = page.url();
const onWorkPage = dyUrl.includes('/creator-micro/home') || dyUrl.includes('/creator-micro/content/manage');
if (!onWorkPage) {
  await dy.navigateToCreatorHome(page);   // 强制导航回 home
}
// 导航后再次校验，仍偏离才 fast-fail
const afterUrl = page.url();
const ok = afterUrl.includes('/creator-micro/home') || afterUrl.includes('/creator-micro/content/manage');
if (!ok) { logger.warn(...); return { ...fast-fail }; }
```
- 先导航再判断，避免"残留子页 → 直接 fast-fail"的 0s 假成功。
- `navigateToCreatorHome` 已有点击导航 + goto 回退 + `pageLoadBehavior` 等待。
- fast-fail 保留作"导航都救不回来"的兜底，不再是常态。
- 验收：抖音日志出现 `Phase1 开始` + `Phase1 完成` + 真实 elapsed（不再是 40ms fast-fail）。

### 第 3 节：恢复登录按钮改造（Bug 3）

**A. 后端新路由** `POST /monitor/accounts/:userId/recover-login`（matrix.ts）
- `prisma.platformAccount.update({ status: 'active', cooldownUntil: 0n })`
- 清该平台所有 flowState（复用 `markLoginRecovered` 的清状态部分）
- `resetSchedulerTimer(windowId, platform)` —— 启动 3s 倒计时
- 返回 `{ message: '已置为已登录并启动监控', userId, platform }`
- 不连浏览器、不检测、不发卡片

**B. `markLoginRecovered` 职责澄清**
- 现有函数（`:764`）只做 DB 重置 + 清 flowState，**不含** `scheduleNext`。保持纯 DB 语义不变。
- 倒计时由新路由显式调 `resetSchedulerTimer`。
- dispatch 入口（`:987-1003`）监控成功后调 `markLoginRecovered`，那里不需要 `scheduleNext`（`reportMonitorComplete` → `scheduleNext` 已在 `:1977` 处理）。

**C. 前端**（`page.tsx:1704-1720`）
- "恢复登录"按钮 `onClick` 改调 `POST /recover-login`（新增 `useRecoverLogin` hook in `useApi.ts`，仿 `useProbeLogin:535`）。
- Toast 改为"已置为已登录，3秒后开始监控"。
- `probe-login` 路由 + `useProbeLogin` hook 保留（"强制刷新"等其他入口可能仍用），不删除。

**D. `probe-login` 路由保留**
- `triggerLoginProbe` 仍被概念复用，"强制刷新登录页"卡片指令可能间接依赖。保留路由，仅前端"恢复登录"按钮改用新路由。

**验收**：
- 点"恢复登录"→ 卡片立即变 active → 倒计时 3s → 3s 后自动跑一轮监控
- 不连浏览器、不发企微卡片
- 若实际未登录，下轮监控 `runKuaishouCheck` 发现失效 → 发一张卡片

## 非目标
- 不改 `LoginTabRegistry.checkLoginState` 通用函数（A0 已在前序计划覆盖内存命中校验）。
- 不改 QR 截图/激活逻辑（A1 已覆盖）。
- 不删除 `probe-login` 路由与 `triggerLoginProbe`。
- 不改小红书/抖音的发卡片策略（仅抖音入口导航修复）。
