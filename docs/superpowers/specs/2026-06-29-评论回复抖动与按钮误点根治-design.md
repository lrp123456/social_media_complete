# 评论回复 scrollCommentArea 抖动 + 按钮误点修复设计

## 背景

抖音评论回复功能存在两个 bug，经过 3 轮修复尝试后仍未解决：

- **Bug 1**：`scrollCommentArea('top')` 在页面已处于顶部时仍执行完整滚动动画（~9.7 秒），造成可见的上下来回抖动。
- **Bug 2**：回复评论时点击了错误的评论 — 目标是 `lqq` 的"很漂亮"，实际回复了 `lqq` 的"22221"。

## 根因分析

### Bug 1：缺少前置边界检查

`scrollCommentArea`（`douyinCrawler.ts:3561`）的 `'top'/'bottom'` 分支在 for 循环内**先滚动再检查**：

```
for (round = 0; round < 8; round++) {
    await cdpSmartScroll(page, [], 3000, dir);   // ← 即使已在顶部也执行
    await sleep(200~500ms);
    const state = await cdpGetDocumentScrollState(page);
    if (scrollY <= 0) break;                      // ← 检查在滚动之后
}
```

日志证据：`round:1 scrollY:0 totalMs:9776` — 仅 1 轮，scrollY 已经是 0，但 `cdpSmartScroll` 的鼠标滚动动画 + CDP wheel scroll fallback 已执行了 9.7 秒，造成可见抖动。

### Bug 2：`:nth-child(N)` 选择器设计性错误 + elementFromPoint 视口外失效

完整因果链（4 步连锁失败）：

1. **`:nth-child(N)` 选择器永远失败**
   - `findRootCommentByUsernameContent`（line 5229）拼接 `sels[si] + ':nth-child(' + (ci + 1) + ')'`
   - `ci` 是 `document.querySelectorAll('div[class*="container-"]')` 返回的扁平数组索引
   - `:nth-child(N)` 的语义是"父元素的第 N 个子元素"
   - **这两者不等价** — 第 27 个 `querySelectorAll` 匹配 ≠ 任何父元素的第 27 个子元素
   - 结果：`containerFound: false`，永远失败

2. **`elementFromPoint` 在视口外返回 null**
   - `findRootCommentByUsernameContent` 返回 `rootMatch.y = 1374`（容器中心）
   - `viewportHeight = 1305`，`y:1374 > 1305` → 坐标在视口外
   - `elementFromPoint(1385, 1374)` 返回 null

3. **`scrollRootIntoView` 距离回退找到错误容器**
   - `elementFromPoint` 返回 null → 回退到"最近容器距离匹配"
   - 距离 `(1385, 1374)` 最近的容器是"22221"的评论（不是"很漂亮"）
   - 把"22221"滚动到视口中央

4. **第二次 `findReplyBtnInContainer` 用视口中心命中错误容器**
   - `centerFallback = {x: vp.w/2, y: vp.h/2}` ≈ `(1385, 652)`
   - `elementFromPoint(1385, 652)` → 命中被错误滚到中央的"22221"容器
   - 找到"22221"的回复按钮 → 点击 → 回复了错误的评论

## 修复设计

### Bug 1 修复：前置边界检查

**文件**：`apps/ts-api-gateway/src/crawlers/douyinCrawler.ts`
**方法**：`scrollCommentArea`（line 3561）
**改动**：在 `'top'/'bottom'` 分支的 for 循环前，加前置边界检查：

```typescript
// 前置检查：已在边界则跳过滚动，避免不必要的动画抖动
const initialState = await HumanActions.cdpGetDocumentScrollState(page);
if (initialState) {
  if (direction === 'top' && initialState.scrollY <= 0) {
    logger.info({ direction, scrollY: initialState.scrollY }, '[scrollCommentArea] already at boundary, skipping');
    return true;
  }
  if (direction === 'bottom' && initialState.scrollY + initialState.clientHeight >= initialState.scrollHeight - 10) {
    logger.info({ direction, scrollY: initialState.scrollY }, '[scrollCommentArea] already at boundary, skipping');
    return true;
  }
}
```

**影响范围**：所有 5 个 `'top'/'bottom'` 调用点（lines 2795, 3631, 4112, 4988, 5108）。不影响 `number` 入参路径。

### Bug 2 修复：同一次 evaluate 内返回回复按钮坐标（方案 A）

**核心思路**：消除 `:nth-child(N)` 选择器和 `elementFromPoint` 二次定位。在 `findRootCommentByUsernameContent` 的 `page.evaluate` 内，找到匹配容器后直接在同一 DOM 上下文中查找回复按钮。

#### 改动 1：增强 `findRootCommentByUsernameContent` 返回值

**文件**：`apps/ts-api-gateway/src/crawlers/douyinCrawler.ts`
**方法**：`findRootCommentByUsernameContent`（line ~5083）

现有返回值：
```typescript
{ x: number, y: number, containerSel: string, isExpanded: boolean, subReplyCountInPage: number }
```

增强后：
```typescript
{
  x: number,           // 容器中心 x（保留，供 level=2 路径使用）
  y: number,           // 容器中心 y（保留，供 level=2 路径使用）
  containerSel: null,  // 不再使用，设为 null
  isExpanded: boolean,
  subReplyCountInPage: number,
  replyBtn: { x: number; y: number } | null  // 新增：回复按钮坐标
}
```

**evaluate 内新增逻辑**（在匹配容器后、return 之前）：

```javascript
// 在已匹配的 container 内查找回复按钮
var replyBtn = null;
var opsAreas = container.querySelectorAll('[class*="operations-"]');
for (var oi = 0; oi < opsAreas.length; oi++) {
  var items = opsAreas[oi].querySelectorAll('[class*="item-"]');
  for (var ri = 0; ri < items.length; ri++) {
    if ((items[ri].textContent || '').trim() === '回复') {
      var r = items[ri].getBoundingClientRect();
      if (r.width > 0 && r.height > 0 && r.top < vh) {
        replyBtn = { x: Math.round(r.left + r.width / 2), y: Math.round(r.top + r.height / 2) };
        break;
      }
    }
  }
  if (replyBtn) break;
}
// 如果回复按钮不可见（可能在视口外），scrollIntoView 后重新查找
if (!replyBtn) {
  container.scrollIntoView({ block: 'center', behavior: 'instant' });
  for (var oi2 = 0; oi2 < opsAreas.length; oi2++) {
    var items2 = opsAreas[oi2].querySelectorAll('[class*="item-"]');
    for (var ri2 = 0; ri2 < items2.length; ri2++) {
      if ((items2[ri2].textContent || '').trim() === '回复') {
        var r2 = items2[ri2].getBoundingClientRect();
        if (r2.width > 0 && r2.height > 0 && r2.top < vh) {
          replyBtn = { x: Math.round(r2.left + r2.width / 2), y: Math.round(r2.top + r2.height / 2) };
          break;
        }
      }
    }
    if (replyBtn) break;
  }
}
return { x, y, containerSel: null, isExpanded, subReplyCountInPage, replyBtn: replyBtn };
```

**关键点**：
- `scrollIntoView` 在 evaluate 内同步执行，操作的是正确的 DOM 元素引用（不是 stale 坐标）
- 两个分支（`isAntiDetectionV2()` + 普通分支）都需要增强

#### 改动 2：改写 `scrollExpandAndFindTarget` level=1 路径

**文件**：`apps/ts-api-gateway/src/crawlers/douyinCrawler.ts`
**方法**：`scrollExpandAndFindTarget`（lines 5022-5065）

现有流程：
```
findRootComment → findReplyBtnInContainer(containerSel, fallback)
  → null → scrollRootIntoView(x, y) → findReplyBtnInContainer(containerSel, centerFallback)
```

新流程：
```
findRootComment (增强版，内含回复按钮查找 + scrollIntoView)
  → replyBtn 有值 → 直接返回 {x: replyBtn.x, y: replyBtn.y}
  → replyBtn 为 null → scrollRootIntoView(x, y) + wait → 重新调用增强版 findRootComment → 返回 replyBtn
```

**伪代码**：
```typescript
if (target.level === 1) {
  const rootMatch = await this.findRootCommentByUsernameContent(page, target, rootContainerSels);
  // ... null check ...
  if (rootMatch.replyBtn) {
    return rootMatch.replyBtn;  // 直接返回，不再调用 findReplyBtnInContainer
  }
  // replyBtn 为 null：scroll into view 后重试
  logger.warn('[Reply::Find] Root found but reply btn not in evaluate, scrolling into view');
  await this.scrollRootIntoView(page, rootMatch.x, rootMatch.y);
  await HumanActions.wait(page, 300, 600);
  // 重新调用增强版 evaluate（此时 root 应已在视口内）
  const retry = await this.findRootCommentByUsernameContent(page, target, rootContainerSels);
  if (retry?.replyBtn) {
    return retry.replyBtn;
  }
  // 仍失败：返回 null
  logger.error({ containerSel: rootMatch.containerSel }, '[Reply::Find] Root found but reply btn still missing');
  return null;
}
```

#### 改动 3：消除的代码

| 代码 | 位置 | 处理 |
|------|------|------|
| `:nth-child(N)` 选择器拼接 | line 5229, 5316 | 设为 `null`，不再拼接 |
| `findReplyBtnInContainer` 调用 | line 5025, 5034 | 删除，替换为 `rootMatch.replyBtn` |
| `elementFromPoint` 5x5 网格回退 | `findReplyBtnInContainer` 内 | 保留函数体，level=1 不再调用 |
| `containerSel` DIAG 诊断块 | `findReplyBtnInContainer` 内 | 保留函数体，level=1 不再触发 |

#### 保留的代码

| 代码 | 原因 |
|------|------|
| `findReplyBtnInContainer` 函数 | 保留，未来可能用于其他场景 |
| `scrollRootIntoView` | 用于 `replyBtn=null` 回退 + level=2 路径 |
| `expandRootRepliesIfNeeded` | level=2 路径不变 |

## 不在本次修复范围

- `expandRootRepliesIfNeeded`（level=2 路径）仍使用 `elementFromPoint` — 风险较低，level=2 场景较少
- `scrollRootIntoView` 本身的 `elementFromPoint` + 距离回退逻辑 — 保留，但 level=1 路径仅在 `replyBtn=null` 时才调用
- `cdpSmartScroll` 性能优化（9.7 秒单次调用）— 独立问题，前置检查可绕过
- 诊断日志清理 — 暂时保留
