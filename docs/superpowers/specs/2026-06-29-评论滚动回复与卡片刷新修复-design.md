# 评论滚动回复失败 与 企业微信卡片刷新失效 — Design Spec

**Date:** 2026-06-29
**Status:** Draft
**Scope:** 修复两个独立 bug —— (1) 抖音评论回复时滚动状态读取对象错误，导致视口外的目标评论被判定"找不到"；(2) 企业微信登录卡片的"强制刷新/F5刷新"按钮点击无效。

## 2. 根因分析（已用 CDP + 容器内 tsx 复现验证）

### 2.1 任务1根因：滚动状态读取对象错误

在窗口 `68a259626bb2c5905ffed8116e9a2a04`（账号1-李瑞鹏，douyin logged_in）的评论管理页 `creator.douyin.com/creator-micro/interactive/comment` 实测：

1. **滚错了容器**。`scrollCommentArea`（`douyinCrawler.ts:3561`）与 `tryExpandMoreAndScroll`（`douyinCrawler.ts:5867`）的"到顶/到底"判定都读 `.douyin-creator-interactive-tabs-content`。实测该容器：
   - `overflow: visible`、`scrollHeight === clientHeight === 2620`、`scrollTop: 0`
   - 即**它根本不滚动**，`st + ch >= sh - 10` 恒成立 → 永远判定"到底"。

2. **真正的滚动面是 `document`（window）**。实测 `bodyScrollH: 3052 > innerH: 1305`，真实滚轮可让 `window.scrollY` 从 0 滚到 1747 到底。当前视频评论仅 17 条、**一次性全量加载，无懒加载接口、无 `hasMore` 分页**（全程只触发 `user_message/notice`，无 `comment/list`）。

3. **`cdpSmartScroll` 本身能滚 document**。实测：
   - `cdpSmartScroll(page, ['body'], 800, 'down')` → scrollY 0→800 ✅
   - `cdpSmartScroll(page, ['.douyin-creator-interactive-tabs-content'], 800, 'down')` → scrollY 800→1600 ✅

   即滚动动作是有效的，**唯一错误的是"到底判定"读取了不可滚动的 `tabs-content`**，导致 `scrollExpandAndFindTarget` 第 1 轮 `tryExpandMoreAndScroll` 返回 `false` → `break` → `findRootCommentByUsernameContent` 在未充分滚动的 DOM 里找不到视口外的目标评论（`top: 1074`，超出初始视口）。

4. 目标评论 `lqq / 很漂亮` 实测**已在 DOM 中**（dump 抽屉/列表时直接命中），所以这是"找到逻辑因滚动不足而漏掉"，不是"评论没加载"。

### 2.2 任务2根因：`prisma.user` 残留（删表 regression）

- commit `efe277f refactor(db): 删除 User 模型与 users 孤儿表` 删除了 `model User`。
- 但 `wechatBotService.ts` 仍残留 **11 处 `prisma.user`**（漏改），`prisma.user === undefined` → 强制刷新（`:778`）、F5刷新（`:822`）分支第一行 `prisma.user.findUnique` 即抛 `Cannot read properties of undefined (reading 'findUnique')`。
- 同文件其他处已改用 `prisma.operator`（`:511/:603`）；`monitorService` 已全面改用 `prisma.platformAccount`（含 `windowId`/`wechatUserid`）——**唯独 wechatBotService 的 11 处漏改**。
- `User.fingerprintWindowId` 字段也已不存在，`PlatformAccount` 通过 `windowId`（外键→`browser_windows.id`）关联，`getBrowser` 需要的是 `browser_windows.external_id`。

**11 处位置**：`wechatBotService.ts` 的 499 / 597 / 663 / 696 / 731 / 737 / 764 / 779 / 823 / 910 / 1209。

## 3. 设计决策

### 3.1 任务1：滚动状态读取收口到 HumanActions（读 document）

**决策**：在 `HumanActions` 新增零 JS 的"读取 document 滚动状态"方法，`scrollCommentArea` 与 `tryExpandMoreAndScroll` 的到顶/到底判定改读该方法返回的 document 状态，不再读 `tabs-content`，也不在 crawler 层裸用 `page.evaluate`。

| 项 | 取值 |
|---|---|
| 读取对象 | `document`（`window` 滚动），不再读 `tabs-content` |
| 读取手段 | CDP `Page.getLayoutMetrics`（零 JS，新增 `cdpClient.getLayoutMetrics` 全量返回） |
| 到顶判定 | `scrollY <= 0` |
| 到底判定 | `scrollY + clientHeight >= scrollHeight - 10` |
| 收口出口 | `HumanActions.cdpGetDocumentScrollState(page)` |
| 滚动动作 | 保留 `cdpSmartScroll`（已验证可滚 document，符合收口规范 §6.1） |

**收口规范一致性**（`project_rules.md` §6.1）：
- 滚动动作继续走 `HumanActions.cdpSmartScroll`（不引入 `window.scrollBy/To`、不引入 `page.mouse.wheel`）。
- 滚动状态读取新增 `HumanActions.cdpGetDocumentScrollState`，内部用 CDP `Page.getLayoutMetrics`（新增 `cdpClient.getLayoutMetrics` 全量返回），**零 JS 执行、不操作 DOM、不触发风控代理**。crawler 层不裸用 `page.evaluate` 读 `window.scrollY`。
- `Page.getLayoutMetrics` 返回 `layoutViewport`（`pageX`/`pageY` = 滚动偏移，`clientWidth`/`clientHeight` = 视口）与 `contentSize`（`height` = 文档总高）。`pageY` 即 `scrollY`。

**兜底（应对未来多评论懒加载场景）**：连续 N 轮滚动后评论 DOM 数无增长且已到底 → 才算穷尽；同时移除 `tryExpandMoreAndScroll` 中"展开更多评论"点击的 `scrollRound % 3 === 0` 限制（实测该按钮在本页不存在，去掉限制无害且更稳健）。

**视口外目标处理**：`findRootCommentByUsernameContent` 命中视口外目标时，复用已有 `scrollRootIntoView`（`:5039`，level1 路径需确认也走该滚动）后再做 `elementFromPoint`/坐标定位，避免视口外匹配失败。

### 3.2 任务2：11 处 `prisma.user` → `prisma.platformAccount` + join 取 externalId

**决策**：参照 `monitorService.ts:417`，把 11 处 `prisma.user.findUnique/findFirst/update` 改为 `prisma.platformAccount`，`fingerprintWindowId` 改为通过 `include: { window: { select: { externalId: true } } }` 取 `browser_windows.external_id` 传入 `getBrowser`，`wechatUserid` 直接取 `platformAccount.wechatUserid`。

| 项 | 取值 |
|---|---|
| 查询 model | `prisma.platformAccount`（`where: { id }`，id 即原 userId） |
| `getBrowser` 入参 | `window.externalId`（join `browser_windows`） |
| `wechatUserid` | `platformAccount.wechatUserid` |
| `status/cooldownUntil/monitoringEnabled` | `platformAccount` 已有同名字段，直接迁移 |
| 参照实现 | `monitorService.ts:417`（`include window`）、`:444/:470`（`user.wechatUserid`） |

字段映射确认（`schema.prisma`）：`PlatformAccount` 含 `wechatUserid`、`status`、`cooldownUntil`、`monitoringEnabled`、`loginStatus`、`windowId`（→`browser_windows.id`→`externalId`）。与原 `User` 用到的字段一一对应。

**两类查询分别处理**（审查发现，不能机械替换）：

| 类别 | 位置 | 原代码 | 迁移方式 |
|---|---|---|---|
| 正向 by id | 499/597/696/731/737/764/779/823/910 | `prisma.user.findUnique({where:{id}, select:{fingerprintWindowId,wechatUserid}})` + `String(user.fingerprintWindowId)` 传 `getBrowser` | `prisma.platformAccount.findUnique({where:{id}, include:{window:{select:{externalId}}}})` + `user.window.externalId` |
| 反向 by wechatUserid | 663、1209 | `prisma.user.findFirst({where:{wechatUserid}, select:{id}})` | `prisma.platformAccount.findFirst({where:{wechatUserid}, select:{id}})`（字段名相同） |

**L663 特例（"回复"入口，含独立旧字段 bug）**：原代码 L663-680 不仅 `prisma.user`，还紧跟一段 `(prisma as any).browserWindow?.findFirst({ where: { userId: user.id, platform }, select: { fingerprintWindowId: true } })`。审查确认旧 `BrowserWindow` schema **从未有 `userId`/`platform`/`fingerprintWindowId` 字段**（commit `efe277f^` 验证），此 `(prisma as any)` 绕类型查询运行时必抛 "Unknown argument" → 被 `.catch` 吞 → `window=null` → "❌ 未找到关联的浏览器窗口"，"回复"功能一直坏。

L663 整段需合并为单次 `platformAccount` 查询（`platformAccount` 同时有 `wechatUserid` 和 `platform` 字段，一步到位）：

```typescript
const acct = await prisma.platformAccount.findFirst({
  where: { wechatUserid: userid, platform },
  select: { id: true, window: { select: { externalId: true } } },
}).catch(() => null);
if (!acct?.window?.externalId) {
  await botManager.sendTextMessage([userid], '❌ 未找到关联的浏览器窗口');
  return;
}
botManager.setPendingReply(commentCid, {
  videoId: awemeId, awemeId, userId: acct.id, windowId: acct.window.externalId, platform,
});
```

L1209（实际回复文本入队）为纯反向取 id，按上表"反向 by wechatUserid"处理即可。

## 4. 实现计划（概要，详细步骤交由 writing-plans）

### 4.1 任务1

1. **新增** `cdpClient.getLayoutMetrics(): Promise<{ layoutViewport: {pageX,pageY,clientWidth,clientHeight}; contentSize: {x,y,width,height} }>`
   - 直接 `this.send('Page.getLayoutMetrics', {})` 返回原始结构（含 `layoutViewport` + `contentSize`）。现有 `getLayoutViewport` 只取 `layoutViewport` 丢了 `contentSize`，故新增全量方法（不改 `getLayoutViewport`，避免影响现有调用方）。
2. **新增** `HumanActions.cdpGetDocumentScrollState(page): Promise<{ scrollY: number; clientHeight: number; scrollHeight: number } | null>`
   - `withCDPContext` → `cdp.getLayoutMetrics()` 取 `layoutViewport.pageY`（=scrollY）、`layoutViewport.clientHeight`、`contentSize.height`（=scrollHeight）。
   - 失败返回 `null`（不静默降级，遵循铁律）。
2. **改** `scrollCommentArea`（`:3587-3604`）：`'top'/'bottom'` 分支的状态读取从 `tabs-content` 的 `scrollTop/scrollHeight/clientHeight` 改为 `cdpGetDocumentScrollState` 返回的 document 状态。到顶 `scrollY<=0`、到底 `scrollY+clientHeight>=scrollHeight-10`。
3. **改** `tryExpandMoreAndScroll`（`:5912-5933`）：同上改读 document 状态；移除"展开更多评论"的 `scrollRound % 3 === 0` 限制；加"连续 N 轮评论数无增长 + 到底才算穷尽"兜底。
4. **不动 `findRootCommentByUsernameContent` 的视口内限制**（审查澄清）：该函数 `:5195` 有 `if (rect.top < vh && rect.bottom > 0 ...)` 只匹配视口内容器——视口外目标在此函数内即返回 `null`，到不了 `:5039` 的 `scrollRootIntoView`（那是在 rootMatch 已找到之后）。主修复（步骤2/3 改读 document 到底判定）修对后，主循环能正常滚动，目标终会进入视口被 `:5195` 命中，**主流程修复成功不依赖改这里**。故本步骤仅记录此约束，不做改动；若后续要支持"先扫全 DOM 再滚入视口"可单列优化，不在本次范围。
5. **测试**：补/改 `__tests__/scrollBounded.test.ts`、`findTargetBudget.test.ts`，断言 document 滚动状态读取与到底判定。

### 4.2 任务2

1. **正向 by id（9 处：499/597/696/731/737/764/779/823/910）**：`prisma.user.findUnique/findFirst/update` → `prisma.platformAccount`，需要 `externalId` 处加 `include:{window:{select:{externalId}}}`，`String(user.fingerprintWindowId)` → `user.window.externalId`，`user.wechatUserid` 保持。
2. **L663 "回复"入口（含旧字段 bug）**：按 §3.2 特例，整段（`prisma.user` + 紧跟的 `browserWindow.findFirst({userId,platform})`）合并为单次 `prisma.platformAccount.findFirst({where:{wechatUserid,platform}, select:{id, window:{select:{externalId}}}})`，删掉 `(prisma as any).browserWindow` 段。
3. **L1209 反向取 id**：`prisma.user.findFirst({where:{wechatUserid},select:{id}})` → `prisma.platformAccount.findFirst({where:{wechatUserid},select:{id}})`。
4. **update 类（731/737/764/910）**：`prisma.user.update({where:{id},data:{status,cooldownUntil,monitoringEnabled}})` → `prisma.platformAccount.update`（同名字段）。
5. 验证：手动触发"强制刷新"/"F5刷新"卡片按钮，确认不再抛 `findUnique` 错误且能刷新登录页、发送新二维码；额外触发"回复 <platform> <awemeId> <commentCid>"，确认不再报"未找到关联的浏览器窗口"。

## 5. 风险与回滚

- **任务1**：`Page.getLayoutMetrics` 的 `contentSize.height` 在某些抖音页面可能不含动态加载内容；若发现 `scrollHeight` 不准，回退用 `HumanActions.safeEvaluate`（收口的 `Runtime.evaluate`，带 `reason` 审计）读 `document.documentElement.scrollHeight`。判定逻辑不变，只换数据源。`scrollHeight` 不准的判定依据为：连续滚动后 `scrollY` 不再增长但 `contentSize.height` 仍 > `scrollY+clientHeight`，此时切换数据源。
- **任务2**：纯字段映射迁移，`platformAccount` 字段已确认齐全；风险低。回滚即还原 11 处 `prisma.user`（但 `User` 表已删，回滚后仍会崩，故无需回滚，必须修对）。
- **两任务独立**，可分别提交、分别验证。

## 6. 验收标准

1. **任务1**：在评论数 > 视口可显示量、目标评论位于视口外的视频上触发回复，能正确滚动 document 找到目标并完成回复；不再出现 `Bottom reached scrollRound=1` 后立即 `Exhausted, no match`。debug 快照显示 `scrollY` 随轮次增长。
2. **任务2**：点击企微卡片"强制刷新登录页"/"F5刷新QR码"，日志不再出现 `Cannot read properties of undefined`，能刷新登录页并发送新二维码卡片；触发"回复 <platform> <awemeId> <commentCid>"不再报"未找到关联的浏览器窗口"（L663 旧字段 bug 一并修复）。
3. 收口规范：crawler 层无新增 `page.evaluate`/`window.scrollBy/To`/`page.mouse.wheel`；所有滚动状态读取经 `HumanActions.cdpGetDocumentScrollState`。
