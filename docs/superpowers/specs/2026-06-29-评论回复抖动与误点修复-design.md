# 评论回复抖动与误点修复设计

## 背景

抖音评论回复功能存在两个持续 bug，经过 3 轮修复尝试仍未解决：

- **Bug 1（滚动抖动）**：`scrollCommentArea('top')` 进入回复流程时页面上下来回抖动
- **Bug 2（点错评论）**：回复目标评论"很漂亮"时，实际回复了同用户的另一条评论"22221"

## 根因分析

### Bug 1：scrollCommentArea 缺少前置边界检查

`scrollCommentArea` 的 `'top'/'bottom'` 分支在 for 循环内**先滚动再检查边界**：

```typescript
for (let round = 0; round < SCROLL_MAX_ROUNDS; round++) {
  await HumanActions.cdpSmartScroll(page, [], SCROLL_BOUNDED_PX, dir);  // 先滚动
  await new Promise(r => setTimeout(r, 200 + Math.random() * 300));
  const state = await HumanActions.cdpGetDocumentScrollState(page);
  if (direction === 'top' && state.scrollY <= 0) { break; }  // 后检查
}
```

即使页面已在顶部（`scrollY:0`），`cdpSmartScroll([], 3000, 'up')` 仍执行完整鼠标滚动动画（含多次 CDP wheel scroll fallback + overshoot），耗时 9.7 秒，造成可见抖动。

日志证据：
```
round:1 direction:"top" scrollY:0 clientHeight:1305 scrollHeight:2041
msg:"[scrollCommentArea] reached top/bottom"
direction:"top" totalMs:9776
```

### Bug 2：:nth-child(N) 选择器设计性错误 + elementFromPoint 视口外失效

**缺陷 1 — `:nth-child(N)` 选择器永远失败**

`findRootCommentByUsernameContent` 在 evaluate 内拼接选择器：

```javascript
var containers = document.querySelectorAll('div[class*="container-"]');
// ci 是 querySelectorAll 返回的扁平数组索引
var containerSel = sels[si] + ':nth-child(' + (ci + 1) + ')';
```

`querySelectorAll` 返回 DOM 树中所有匹配的后代元素（扁平列表），而 `:nth-child(N)` 表示「父元素的第 N 个子元素」。第 N 个 `querySelectorAll` 匹配 ≠ 父元素的第 N 个子元素。因此 `containerFound` 永远为 false。

日志证据：
```
containerSel: "div[class*=\"container-\"]:nth-child(27)"
diag: {"containerFound":false,"viewportHeight":1305}
```

**缺陷 2 — elementFromPoint 在视口外返回 null**

`findRootCommentByUsernameContent` 找到"很漂亮"时返回中心坐标 `y:1374`，但 `viewportHeight:1305`，即坐标在视口下方。后续 `elementFromPoint(1385, 1374)` 返回 null。

**缺陷 3 — scrollRootIntoView 距离回退找到错误容器**

`scrollRootIntoView` 中 `elementFromPoint` 返回 null 后，回退到「距离最近的容器」匹配，找到了错误的容器（"22221"的评论），将其滚动到视口中央。

**缺陷 4 — 第二次 findReplyBtnInContainer 用视口中心命中错误容器**

第二次调用使用 `centerFallback = {x: vp.w/2, y: vp.h/2}`，`elementFromPoint` 在视口中心命中了被错误滚到中央的"22221"容器，返回了"22221"的回复按钮坐标。

完整因果链：
```
findRootComment → 返回 y:1374（视口外）
  → findReplyBtnInContainer(:nth-child(27)) → containerFound:false → elementFromPoint(1385,1374) → null
  → scrollRootIntoView(1385,1374) → elementFromPoint → null → 距离回退 → 滚到错误容器"22221"
  → findReplyBtnInContainer(:nth-child(27), centerFallback) → elementFromPoint(vp中心) → 命中"22221"
  → 点击"22221"的回复按钮 → 回复错误评论
```

## 设计

### Bug 1 修复：scrollCommentArea 前置边界检查

在 for 循环前读取 `cdpGetDocumentScrollState`，如果已在边界则直接返回，不触发任何滚动动画。

**改动位置**：`douyinCrawler.ts` `scrollCommentArea` 方法，`'top'/'bottom'` 分支，for 循环前。

**伪代码**：
```typescript
// 前置边界检查：已在顶部/底部时跳过滚动，避免抖动
const initialState = await HumanActions.cdpGetDocumentScrollState(page);
if (initialState) {
  if (direction === 'top' && initialState.scrollY <= 0) {
    logger.info({ direction, scrollY: initialState.scrollY, ... }, '[scrollCommentArea] already at boundary, skipping');
    return true;
  }
  if (direction === 'bottom' && initialState.scrollY + initialState.clientHeight >= initialState.scrollHeight - 10) {
    logger.info({ direction, scrollY: initialState.scrollY, ... }, '[scrollCommentArea] already at boundary, skipping');
    return true;
  }
}
```

**影响范围**：douyinCrawler 中 5 个 `'top'/'bottom'` 调用点均受益。`number` 入参路径不受影响。

### Bug 2 修复：增强 findRootCommentByUsernameContent，同一次 evaluate 内返回回复按钮坐标

**核心思路**：消除 `:nth-child(N)` 选择器和 `elementFromPoint` 回退，在找到匹配容器的同一个 evaluate 内直接查找该容器的回复按钮。容器 DOM 引用在 evaluate 内始终有效，不存在坐标过期问题。

**改动 1 — 增强 findRootCommentByUsernameContent 的 evaluate 逻辑**

在两个分支（`isAntiDetectionV2()` 和普通 `page.evaluate`）中，找到匹配容器后增加回复按钮查找：

```javascript
// 现有逻辑：找到匹配的 container（username + text 匹配通过）
// ── 新增：在该 container 内查找回复按钮 ──
var replyBtn = null;
var opsAreas = container.querySelectorAll('[class*="operations-"]');
for (var oi = 0; oi < opsAreas.length; oi++) {
  var items = opsAreas[oi].querySelectorAll('[class*="item-"]');
  for (var ri = 0; ri < items.length; ri++) {
    if ((items[ri].textContent || '').trim() === '回复') {
      var r = items[ri].getBoundingClientRect();
      if (r.width > 0 && r.height > 0 && r.top < vh) {
        replyBtn = { x: Math.round(r.left + r.width / 2), y: Math.round(r.top + r.height / 2) };
        break;
      }
    }
  }
  if (replyBtn) break;
}
// 如果回复按钮不可见（可能在视口外），scrollIntoView 后重新查找
if (!replyBtn) {
  container.scrollIntoView({ block: 'center', behavior: 'instant' });
  for (var oi2 = 0; oi2 < opsAreas.length; oi2++) {
    var items2 = opsAreas[oi2].querySelectorAll('[class*="item-"]');
    for (var ri2 = 0; ri2 < items2.length; ri2++) {
      if ((items2[ri2].textContent || '').trim() === '回复') {
        var r2 = items2[ri2].getBoundingClientRect();
        if (r2.width > 0 && r2.height > 0 && r2.top < vh) {
          replyBtn = { x: Math.round(r2.left + r2.width / 2), y: Math.round(r2.top + r2.height / 2) };
          break;
        }
      }
    }
    if (replyBtn) break;
  }
}
```

返回值增加 `replyBtn` 字段：
```typescript
{
  x: number;           // 根评论中心坐标（仍用于 level=2 路径）
  y: number;
  containerSel: null;  // 不再使用，置 null
  isExpanded: boolean;
  subReplyCountInPage: number;
  replyBtn: { x: number; y: number } | null;  // 新增
}
```

**改动 2 — 简化 scrollExpandAndFindTarget level=1 路径**

现有代码（lines 5022-5065）：
```typescript
if (target.level === 1) {
  const fallback = { x: rootMatch.x, y: rootMatch.y };
  let replyBtn = await this.findReplyBtnInContainer(page, rootMatch.containerSel, fallback);
  if (!replyBtn) {
    await this.scrollRootIntoView(page, rootMatch.x, rootMatch.y);
    const vp = await page.evaluate(() => ({ w: window.innerWidth, h: window.innerHeight }));
    const centerFallback = { x: Math.round(vp.w / 2), y: Math.round(vp.h / 2) };
    replyBtn = await this.findReplyBtnInContainer(page, rootMatch.containerSel, centerFallback);
  }
  // ... 诊断 + return
}
```

新代码：
```typescript
if (target.level === 1) {
  // 方案 A：findRootCommentByUsernameContent 已在 evaluate 内查找回复按钮
  if (rootMatch.replyBtn) {
    logger.info({ elapsedMs: Date.now() - startT0 }, '[Reply::Find] Root reply btn located');
    return rootMatch.replyBtn;
  }
  // replyBtn 为 null：按钮在视口外且 evaluate 内 scrollIntoView 后仍未找到
  // 回退：scrollRootIntoView + 重新调用 findRootCommentByUsernameContent
  logger.warn('[Reply::Find] Reply btn not found in evaluate, scrolling into view and retrying');
  await this.scrollRootIntoView(page, rootMatch.x, rootMatch.y);
  await HumanActions.wait(page, 300, 600);
  const retryMatch = await this.findRootCommentByUsernameContent(page, target, rootContainerSels);
  if (retryMatch?.replyBtn) {
    logger.info({ elapsedMs: Date.now() - startT0 }, '[Reply::Find] Root reply btn located (retry)');
    return retryMatch.replyBtn;
  }
  // 仍找不到：返回 null
  logger.error({ containerSel: rootMatch.containerSel }, '[Reply::Find] Root found but reply btn still missing after retry');
  return null;
}
```

**改动 3 — 消除的代码**

- `:nth-child(N)` 选择器拼接（2 处 evaluate 分支）
- level=1 路径对 `findReplyBtnInContainer` 的 2 处调用
- `elementFromPoint` 5x5 网格回退（level=1 不再需要）

**保留的代码**

- `findReplyBtnInContainer` 函数本体（保留，level=1 不再调用但不删除）
- `scrollRootIntoView`（用于 replyBtn=null 回退和 level=2 路径）
- `expandRootRepliesIfNeeded`（不变，level=2 路径不在本次修复范围）
- 诊断日志（DIAG，暂时保留便于观察）

## 不在本次修复范围

- `expandRootRepliesIfNeeded` 中使用 `elementFromPoint` 的 level=2 路径（风险较低，用户未报告问题）
- `scrollRootIntoView` 本身的距离回退逻辑（仅在 replyBtn=null 回退路径使用，增强版 evaluate 内已优先用 `container.scrollIntoView`）
- `cdpSmartScroll` 的性能优化（9.7 秒单次滚动调优，前置检查已避免不必要的调用）
- 诊断日志（DIAG）清理（后续观察确认稳定后删除）
