## 一、项目现状分析

### 1.1 现有实现概况

通过对项目代码的全面审查，当前各平台评论回复功能的实现状态如下：

| 功能模块 | 抖音 | 快手 | 视频号（参考） |
|---------|------|------|-------------|
| 评论管理导航 | ✅ 已实现 | ✅ 已实现 | ✅ 已实现 |
| 评论列表获取 | ✅ 已实现 | ✅ 已实现 | ✅ 已实现 |
| 子回复展开 | ✅ 已实现 | ✅ 已实现 | ✅ 已实现 |
| **回复评论操作** | ✅ 已实现 | ❌ **未实现** | ✅ 已实现 |
| 回复选择器配置 | ✅ 已配置 | ❌ **缺失** | ✅ 已配置 |

### 1.2 关键发现

**快手回复功能完全缺失**：`monitorService.ts` 的 `executeReplyAction()` 函数中仅有 `douyin` 和 `tencent` 的处理分支，没有 `kuaishou` 分支。同时 `wechatBotService.ts` 中回复任务的 `platform` 字段硬编码为 `'douyin'`，不支持快手平台。

`selectors.json` 中快手部分也缺少回复按钮、回复输入框、发送按钮等关键选择器配置。

### 2.3 回复评论完整操作流程 `[浏览器验证]`

#### 前置条件

- 已登录抖音创作者中心
- 浏览器页面位于 `creator.douyin.com` 域名下
- 已进入评论管理页面并加载了评论列表

#### 步骤 1：导航到评论管理页面

- **操作**：导航至 `https://creator.douyin.com/creator-micro/interactive/comment`
- **或**：点击左侧菜单 `#douyin-creator-master-menu-nav-comment_manage_new`
- **等待时间**：2000-3000ms（页面加载及评论数据请求）
- **验证**：页面标题显示"评论管理"，评论列表已渲染

#### 步骤 2：定位目标评论

- **操作**：遍历 `div.container-sXKyMs` 元素，通过 `div.comment-content-text-JvmAKq` 的文本内容匹配目标评论
- **选择器**：`[class*="container-sXKyMs"]`
- **注意**：当前版本 **没有** `data-cid` 属性，无法通过评论 ID 直接定位，需要通过文本匹配或 API 拦截获取映射关系

#### 步骤 3：点击回复按钮

- **目标元素**：在目标评论的 `div.operations-WFV7Am` 容器中，找到文本为"回复"的 `div.item-M3fSkJ`
- **CSS 选择器**：`[class*="operations"] [class*="item-"]` 中 `textContent.trim() === '回复'` 的元素
- **备选选择器**：`text=回复`（全局文本匹配，需限定在当前评论容器内）
- **触发方式**：**始终可见**，不需要 hover 即可看到回复按钮
- **点击后等待**：500-1000ms（输入框展开动画）

#### 步骤 4：输入回复内容

- **输入框位置**：点击回复按钮后，在该评论下方展开回复区域
- **回复区域容器**：`div.reply-content-obcMk0`
- **输入框选择器**：`div.reply-content-obcMk0 div[contenteditable="true"]`
- **输入框 class**：`input-d24X73`
- **Placeholder**：`"回复 {用户名}："`（如 "回复 lqq："）
- **输入方式**：逐字输入（CDP keyPress），50-150ms/字
- **字数限制**：待验证

> **注意**：页面上有两个 `contenteditable="true"` 输入框：一个是顶部的主评论输入框（placeholder: "有爱评论，说点好听的~"），另一个是点击回复后出现的回复输入框（placeholder 含"回复"字样）。操作时需选择正确的输入框。

#### 步骤 5：发送回复

- **发送按钮选择器**：`button.douyin-creator-interactive-button-primary:not([class*="disabled"])` 中 `textContent === '发送'` 的按钮
- **完整 HTML**：
  ```html
  <button class="douyin-creator-interactive-button
                 douyin-creator-interactive-button-primary
                 douyin-creator-interactive-button-size-small">
    <span class="douyin-creator-interactive-button-content">发送</span>
  </button>
  ```
- **空内容时状态**：按钮带 `douyin-creator-interactive-button-primary-disabled` class，`aria-disabled="true"`，**不可点击**
- **有内容时状态**：按钮 class 变为 `douyin-creator-interactive-button-primary`，**可点击**
- **取消按钮**：`button.douyin-creator-interactive-button-tertiary` + 文本"取消"，用于关闭回复输入框
- **点击后等待**：2000-3000ms（网络请求 + 页面更新）

#### 步骤 6：验证发送结果

- **成功标识**：
  - 页面出现 Toast 提示：**"回复成功"**（class: `douyin-creator-interactive-toast`）
  - 回复输入框自动关闭
  - 新回复出现在目标评论下方，包裹在 `div.reply-list-QwXCb_` 容器中
  - 新回复带有红色 **"作者"** 标识（`span.douyin-creator-master-badge`）
- **失败标识**：待验证（未触发过失败场景）

### 2.5 API 接口信息 `[浏览器验证]`

**参数说明：**

| 参数 | 类型 | 说明 |

| `aweme_id` | string | 视频 ID（作品 ID） |

| `reply_to_comment_ids` | string | 被回复的评论 ID |
| `channel_id` | number | 频道 ID（固定值 618） |
| `aid` | number | 应用 ID（固定值 2906，URL 参数） |
| `device_platform` | string | 设备平台（固定值 webapp，URL 参数） |

#### 评论列表 API（代码推断 + 抓包确认）

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 评论列表 | GET | `/aweme/v1/web/comment/list/select` | 获取视频评论列表 |

**请求特征**：需要携带安全签名参数 `x-bogus`、`mstoken`，由前端 JS 动态计算，难以直接模拟。项目通过 API 拦截（`page.on('response')`）捕获浏览器自动发出的请求来解析评论数据。

### 2.6 页面技术栈特征 `[浏览器验证]`

| 技术项 | 详情 |
|--------|------|
| 前端框架 | React |
| UI 组件库 | **Semi Design**（字节跳动自研）—— 已验证，页面中大量使用 `douyin-creator-interactive-*` 前缀组件和 `x-semi-prop` 属性 |
| CSS 方案 | CSS Modules 哈希类名 + Semi Design 组件库样式 |
| CSS 前缀 | 业务: `douyin-creator-interactive-*`、`douyin-creator-master-*`；哈希: `container-sXKyMs`、`operations-WFV7Am` 等 |
| 类名模式 | CSS Modules 生成的哈希后缀（如 `-sXKyMs`、`-WFV7Am`）附加在语义化前缀后 |
| 图标系统 | `dux-icon` + SVG 内联图标 |
| 类型系统 | TypeScript |

**选择器稳定性提示**：CSS Modules 哈希后缀（如 `container-sXKyMs` 中的 `-sXKyMs`）在每次构建时可能变化。建议优先使用以下策略：

1. **文本内容匹配**：`text=回复`、`text=发送` — 最稳定
2. **Semi Design 组件 class**：`douyin-creator-interactive-button-primary` — 较稳定
3. **属性选择器**：`[contenteditable="true"]` — 稳定
4. **哈希 class 的模糊匹配**：`[class*="container-"]`、`[class*="operations"]` — 中等稳定性（前缀不变，后缀可能变）

### 2.7 现有代码实现参考 `[代码推断]`

```typescript
// 文件：monitorService.ts (executeReplyAction 函数)
// 抖音平台回复流程

try {
  // 1. 导航到创作者中心 & 评论管理
  await douyinCrawler.navigateToCreatorHome(page);
  await douyinCrawler.navigateToCommentManage(page);

  // 2. 打开作品选择抽屉 & 选择视频
  await douyinCrawler.openSelectWorkDrawer(page);
  await douyinCrawler.findAndClickVideoInDrawer(page, replyData.videoId, '');

  // 3. 定位目标评论（通过容器选择器 + 文本匹配）
  const containers = await HumanActions.queryElementsWithInfo(
    page, '[class*="container-sXKyMs"]'
  );

  // 4. 点击"回复"按钮
  await HumanActions.cdpClickByText(page, '回复');

  // 5. 聚焦输入框 & 逐字输入
  await HumanActions.cdpClick(page, 'div[contenteditable="true"]');
  for (const char of replyData.text) {
    await HumanActions.cdpKeyPress(page, char, char, char.charCodeAt(0));
    await HumanActions.wait(page, 50, 150);
  }

  // 6. 点击发送
```

## 六、模拟人类行为的关键要求

### 6.1 鼠标移动

- 不要瞬间移动到目标位置，需要有移动轨迹
- 移动过程中加入轻微的随机偏移
- 到达目标后等待 200-500ms 再点击

### 6.2 键盘输入

- 逐字输入，每字间隔 50-150ms
- 偶尔加入更长的停顿（模拟思考）
- 不要使用粘贴方式一次性输入

### 6.3 点击行为

- 点击前先将鼠标移动到目标元素上方
- 等待 200-500ms 后执行点击
- 点击后等待元素响应（300-500ms）

### 6.4 操作间隔

- 每个主要操作之间等待 500-2000ms
- 加入随机的额外等待时间
- 避免过于规律的操作节奏
